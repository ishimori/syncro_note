## 1. 設計原則（最重要・ここが今回の結論）

実機ベンチ（[DD-002 結果](../DD/DD-002/結果.md)）と多観点のアーキ精査から、**「会議中の逐次LLM整形を議事録の真実源（source of truth）にする」当初設計（要件 File 2）は、速度と信頼性の両面で破綻する**と判断した。中核設計を次のとおり再フレームする。

### 原則1: Whisper の確定テキスト即表示が「主役」
- 画面に最初に出るのは **Whisper が確定した文字起こしテキスト**。遅延ほぼ0・幻覚0・意味改変なし。これが議事録の**真実源**。
- LLMはこのテキストを**上書きしない**。

### 原則2: ケバ取りは「機械処理」を第一選択
- 「えーっと/あの/まあ」等のフィラー除去は **辞書＋正規表現**で十分かつ確実（LLM不要・幻覚0・即時）。
- LLMをフィラー除去の主役にしない。

### 原則3: LLM整形は「非ブロッキングの追い上げレイヤ」（任意）
- LLMによる清書（語尾の整え・専門用語補正等）は、**確定テキストとは別レイヤ**で後追い表示する。遅れても主役の表示は崩れないため、**遅延が事故にならずUX問題に格下げ**される。
- これにより「12.7/9.2 tok/s で発話に追いつくか？」という問い自体が消える（主役が0遅延だから）。

### 原則4: 深い構造化は「終了後バッチ」
- 要約・決定事項抽出・TODO化・話者名寄せの確定などの重い処理は、**会議終了後に全タイムラインを特大モデルで一括再生成**する。会議中を軽く割り切るコストはここで回収されるため実質ゼロ。

### 原則5: 重い推論はホットパスから外す
- 話者分離（diarization）のクラスタリング等のCPU重処理は**リアルタイム経路に置かない**。会議中は仮IDで流し、確定は人間の手修正（要件 File 3 T4）＋終了後オフライン処理に委ねる。

### 原則6: CPUでは STT と LLM を時間分離（既定）
- 本機は DDR5 共有の**メモリ帯域律速**。Whisper と Ollama を同時実行するとスループットが **16〜51% 劣化**しうる。**既定は時間分離（whisper↔LLM を交互）**とし、真の並列は実測で正当化できた範囲だけ採用する。

## 2. 全体アーキテクチャ

### 2.1 段階的進化（ロードマップ準拠）
```
[評価期] Python CLI/スパイク
  音声 → faster-whisper → (辞書ケバ取り) → Ollama要約 → Markdown
   ↓ ホットパス移植・リアルタイム要件が固まったら
[製品期] Tauri アプリ（本設計書の対象）
  cpal(Rust) → ringbuf → VAD → whisper-rs → [確定即表示] ┬→ 辞書ケバ取り(即)
                                                          └→ LLM追い上げ(任意/非同期)
  会議終了 → 全タイムライン＋資料 → 特大MoEで清書バッチ → SQLite永続化
```

### 2.2 リアルタイム・タスクトポロジ（Tauri / Tokio）
```
                 CancellationToken（全タスクにclone）
[OSオーディオRTスレッド] cpal data callback ──push f32──▶ ringbuf SPSC (lock-free)
   ※tokioタスクではない。alloc/ログ/ロック禁止。満杯は戻り値を見て drop＋カウンタ
   （音声欠落は許容・回復可能。オーディオデバイスの stall は不可）

T_audio   tokio::spawn     ringbuf consumer 所有→16k mono化→VAD(無音0.5〜1s / 最大8〜12s)
                           →AudioChunk{seq} ──mpsc bounded(8)──▶  満杯時=隣接チャンクmerge
T_stt     spawn_blocking   whisper-rs(CPUバウンドC++)。専用 blocking プール, n_threads=4
   (loop)                  →RawSegment{seq} ──mpsc bounded(4)──▶  満杯時=同話者連結 or 素通し劣化
                           ★確定テキストを即 window.emit("raw-segment")  ← 表示の主役・真実源
T_clean   軽量2系統        (a) 機械ケバ取り: 辞書+正規表現（同期・幻覚0・即 emit）
                           (b) 任意 LLM追い上げ: single-flight FIFO・非ブロッキング
                              reqwest stream /api/chat ──token──▶ emit("refined-token")
T_state   tokio::spawn     ★ACTOR。SessionState 単独所有。select!{ mpsc<SessionCmd> + cancel }
(Tauri Cmd) update_speaker_mapping / add_human_memo / finish ──SessionCmd──▶ T_state
T_persist                  SQLite(WAL) へ非同期バッチ flush（ホットパスをI/Oで止めない）
finish:   cpal停止 → cancel() → JoinSet で drain → keep_alive:0 で軽量退避
          → /api/ps 空確認 → 特大MoEロード（清書バッチへ）
```

## 3. リアルタイムパイプライン詳細設計

### 3.1 主要データ構造（Rust 擬似コード）
```rust
type Seq = u64;
struct AudioChunk   { seq: Seq, samples: Arc<[f32]>, t_start_ms: u64, t_end_ms: u64 } // 16k mono f32
struct RawSegment   { seq: Seq, speaker_id: u16, text: String, t_start_ms: u64 }       // whisper生
struct CleanSegment { seq: Seq, chunk_id: Uuid, speaker_id: u16, text: String, t_ms: u64 }

// 表示・永続化の単位。表示名は speaker_id から都度導出（文字列を埋め込まない）
enum TimelineElement {
    AiTranscription { id: Uuid, seq: Seq, speaker_id: u16, t_ms: u64, text: String }, // 確定生テキスト=不可変
    HumanMemo       { id: Uuid, t_ms: u64, text: String },                            // File 4: t_msで挿入
}

// confirmed（人間確定）が常に ai_guess（AI推測）に勝つ
struct SpeakerMap { confirmed: HashMap<u16,String>, ai_guess: HashMap<u16,String> }
impl SpeakerMap {
    fn display(&self, id: u16) -> String {
        self.confirmed.get(&id).or(self.ai_guess.get(&id)).cloned()
            .unwrap_or(format!("Speaker_{id}"))
    }
}

struct PromptCtx { system_static: Arc<str>, speaker_map: Arc<SpeakerMap> } // 機械ケバ取りは window 不要

struct SessionState {
    meeting_id: Uuid,
    participants: Vec<Participant>, vocab: Vec<String>, agenda: String,
    speaker_map: SpeakerMap,
    timeline: Vec<TimelineElement>,   // 単一の真実源・順序保証
    system_static: Arc<str>,          // 参加者/アジェンダ/語彙/確定話者を畳んだ固定プレフィクス
    cancel: CancellationToken,
}

enum SessionCmd {
    SnapshotPromptCtx { speaker_id: u16, reply: oneshot::Sender<PromptCtx> },
    AppendRaw     { seg: RawSegment },                    // ★Whisper確定を即 timeline へ（真実源）
    AppendClean   { seg: CleanSegment },                  // 整形は別フィールド。原文は不可変
    UpdateSpeaker { speaker_id: u16, real_name: String }, // File 3: 人間上書き → watch 送出
    SetAiGuess    { speaker_id: u16, name: String },
    AddHumanMemo  { text: String, t_ms: u64 },            // File 4: t_ms で挿入
    Finish        { reply: oneshot::Sender<Vec<TimelineElement>> }, // 全 timeline を清書バッチへ
}
```

### 3.2 チャネル設計
| 区間 | 種別 | 容量 | 満杯時の方針 |
|---|---|---|---|
| cpal cb → audio | **ringbuf SPSC**（lock-free） | 数秒分（16k/f32=64KB/s） | push 戻り値を必ず確認し drop＋カウンタ。`push_overwrite` は使わない |
| audio → whisper | tokio mpsc **bounded(8)** | AudioChunk | 隣接音声を merge（タスクを増やさない） |
| whisper → clean/llm | tokio mpsc **bounded(4)** | RawSegment | 同話者を連結、深刻なら LLM をバイパスし生テキスト即表示（unrefined フラグ） |
| Tauri Cmd → actor | tokio mpsc bounded(64) | SessionCmd（低頻度制御系） | — |
| actor → 表示名読取 | **watch&lt;Arc&lt;SpeakerMap&gt;&gt;** | latest-value | lock-free 最新値読取 |
| actor snapshot 応答 | **oneshot** | PromptCtx | — |
| 全タスク停止 | **CancellationToken** | — | `select!{ _ = cancelled() => break }` |

> `broadcast` はデータ経路に使わない（lossy / lagging consumer は順序付き議事録に不適）。
> トークンのUI送出は Rust チャネルではなく **Tauri `window.emit`**。順序は `seq` で担保。

### 3.3 状態管理 — アクターパターンを採用（`Arc<Mutex>` は不可）
- 理由: `SpeakerMap`・`timeline` は毎LLM呼び出しで読まれ、Tauri Command（話者修正・メモ追記）で非同期に書かれる。`std::Mutex` を**数秒のLLM `.await` を跨いで保持すると実行器ブロック/デッドロック**、`tokio::Mutex` でも**話者修正の瞬間にUIが生成中フリーズ**する。
- 採用: **`T_state` タスク1個が `SessionState` を単独所有**し、全アクセスを `SessionCmd` enum（mpsc）＋ `oneshot` 返信に**直列化**。これで「人間確定 > AI推測」（File 3）と「メモの timestamp 挿入」（File 4）が**単一直列点で競合なく**強制される。
- LLMタスクは **安い snapshot（tiny window の clone ＋ `Arc<SpeakerMap>`）を取得してからロック無しで長い `.await`** に入る。許容する `Mutex`/`Atomic` は「ドロップフレーム計数」と「遅延ゲージ」だけ。

### 3.4 バックプレッシャ / 順序 / キャンセル / コア配分
- **バックプレッシャ（一級設計要素）**: 全チャネル bounded。詰まったら「同話者連結」→さらに深いと「**LLM整形をバイパスし生テキストを即表示**」。原文表示が主役なので劣化＝整形の遅れだけ、議事録は壊れない。遅延秒数をUIにメトリクス表示。
- **順序**: live の LLM 段は **single-flight FIFO**（Ollama は直列。並列化は順序破壊リスクのみ）。VAD/whisper 時点で `seq:u64` 採番、emit に `(chunk_id, seq, token_index)`、フロントは seq→token_index 順、`is_final` でクローズ。
- **話者一括置換（File 3 §3.2）**: **表示名キーの差し替え（O(1) 状態更新 → Vue `computed` で再導出）**であって、レンダリング済み文字列の find-replace ではない。確定テキストは immutable、生成中チャンクは別バッファに隔離 → 進行中トークンとのちらつき・巻き戻りを原理的に排除。
- **キャンセル**: セッション毎 `CancellationToken`。finish 手順 = **cpal停止 → cancel() → JoinSet で drain → keep_alive:0 で軽量退避 → /api/ps 空確認 → 特大MoEロード**。
  - 注意: client の stream 中断で **Ollama サーバ側の生成が即停止する保証はない**（既知 issue）。`num_predict` 上限＋`keep_alive:0`＋`/api/ps` でアイドル確認し、浪費を有界化する。
- **コア配分（8C/16T, Ryzen 7 PRO 8840HS）**: `OLLAMA_NUM_PARALLEL=1` / `OLLAMA_MAX_LOADED_MODELS=1`、whisper `n_threads=4`（専用 blocking プール。tokio 汎用プールに無制限 spawn させない＝semaphore/rayon で制限）、Ollama `num_thread≈3〜6`、cpal+reactor+webview に 1〜2 予約。**合計が物理8コアを超えない**（超えるとスラッシング）。全スレッド数は config 化しロードマップ §5 指標で実機チューニング。
- **emit 頻度**: 1トークン毎の `window.emit` はIPC/JSON 過多 → **数十ms or 数文字でコアレス**して 1/5〜1/10 に。
- **永続化**: WAL 有効 SQLite へ別タスクでバッチ flush。ホットパスをディスクI/Oで stall させない。

## 4. LLM 運用設計（Ollama）

### 4.1 モデル選定（DD-002 実測ベース）
| 用途 | 採用 | 根拠 |
|---|---|---|
| **リアルタイム整形（入れる場合）** | **`qwen3:8b`** | TTFT 0.39秒が最速（体感を最も左右）。prefill 5231 tok/s で固定プレフィクスがほぼ無料。密8Bは日本語固有名詞/専門用語補正に強い |
| リアルタイムのフォールバック | `gemma4:e4b`（11.6 tok/s） | 併走で8bのunder-load tok/sが落ちすぎる場合のスループット余裕枠 |
| **終了後バッチ清書** | **`gemma4:26b`（MoE活性4B, 12.6 tok/s）** | 「26B級品質を4B級速度で」を本機で実証。比較候補 `Qwen3-30B-A3B`（実在MoE, Q4≈18〜20GB） |
| live で不可 | `qwen3:14b`（4.7 tok/s） | 連続発話で完全に追いつかず、帯域競合で真っ先に破綻 |

> 最も堅実な構成は **「会議中は LLM整形を真実源にせず、ケバ取りは辞書/正規表現＋任意の LLM追い上げ」**。この場合 live モデルは無くても成立し、入れるなら `qwen3:8b`。

### 4.2 リアルタイム整形プロンプト方針（採用する場合）
- タスクを**ケバ取り＋人名/専門用語置換のみ**に拘束。「要約するな」「元の意味を変えるな」「整形後テキストだけ出力」を明示。
- `num_predict`（出力上限）で**出力が入力長を超えない**よう固定（ballooning 防止）。`temperature` 低め、`repeat_penalty` で繰り返し抑制、適切な `stop`。
- **コンテキストは原則0**（現在の発話単体＋固定システムプレフィクス）。文脈依存補正は終了後バッチに委譲。
- Gemma 系は thinking 非対応。qwen3 系は **think=false** で運用。

### 4.3 モデル切替（会議中↔終了後）
- `OLLAMA_MAX_LOADED_MODELS=1` でスケジューラの **evict-and-wait** に乗せる（唯一の真の直列保証）。
- 終了時: 軽量モデルへ `keep_alive:0` を投げる → **`/api/ps` が空を確認** → 特大MoEをロード。二重常駐によるOOM/スワップ窓を避ける。
- `keep_alive` は env だけでなく**毎リクエスト body に明示**（live=`-1` 常駐 / 切替=`0`）。既定5分やWindowsの早期unload挙動に振り回されない。
- 長コンテキスト時の節約に `OLLAMA_KV_CACHE_TYPE=q8_0`（メモリ半減・劣化ほぼなし）。スライディングwindowを使う場合は固定プレフィクスを先頭に置き prefix キャッシュを効かせる。

## 5. 話者分離（Diarization）設計 — whisper 非依存

> **重要訂正**: whisper.cpp / whisper-rs に汎用の話者分離機能は**無い**。`--diarize` はステレオL/R分離のみ（モノラルマイクで無効）、`tinydiarize(-tdrz)` は**英語 small.en 専用で「話者交代マーカー」のみ**（安定した話者ID付与も再識別も日本語対応も無し）。要件 File 3 §2.1 の前提は成立しない。

- **会議中**: 話者は**仮ID（Speaker_0/1…）または未確定で流す**。重いクラスタリングはホットパスに置かない。人間が UI で確定（File 3 T4）→ `UpdateSpeaker` で以降を自動補正。
- **話者IDを機械生成する場合の推奨スタック（whisper 非依存・言語非依存・CPU可）**:
  Silero VAD → **ONNX 話者埋め込み（ECAPA-TDNN / 3D-Speaker 系, `ort` crate）** → クラスタリング。
- **終了後（推奨）**: 音声を保存し、**オフライン diarization（pyannote.audio / sherpa-onnx）で全区間を一括ラベリング** → 人間が名寄せ確定。会議中のリアルタイム負荷を負わずに精度を取る。
- 評価期は Python で `faster-whisper + (pyannote / sherpa-onnx)` の CPU 実行安定度を先に測る（DD-004）。
- `participants.voice_profile`（要件 File 1）は埋め込み識別には使えないため役割を再定義（人間の名寄せヒント表示用）。

## 6. 終了後バッチ清書設計

- 入力統合: 会議基本情報＋事前資料 `extracted_text`＋**人間メモ（最優先・全文保持）**＋確定話者名付き全タイムライン。
- モデル: `gemma4:26b`（または `Qwen3-30B-A3B`）。出力 2000〜4000tok 規模なら**実 3〜6 分**かかりうる（活性3〜4B≈12tok/s）。UIの所要時間表示は**実測に合わせる**（「1〜2分」は過小）。
- 長時間会議は **map-reduce 要約**（アジェンダ/時間ブロック単位で要約→統合）で `num_ctx` 超過を回避。人間メモは要約段で切り捨てない。
- 出力 Markdown を `meetings.final_minutes` に保存し `status='completed'`（要件 File 5 §3.2）。

## 7. スループット計算（実現性の根拠）

- **日本語トークン需要**: 本機トークナイザ実測 **≈1.5 tok/文字**（100字=148tok）。発話速度 400〜500字/分 ≒ **入力到着 約10〜12.5 tok/s 相当**。
- **LLMで原文を上書き（当初 File 2 方式）**: 出力≈入力量を再生成するため約10〜12.5 tok/s が必要。実測 8b=8.9 / e4b=11.6 / 14b=4.7 → **連続発話で不足〜拮抗、バックログが線形増大**。最悪遅延（本機 tok/s）: 100字チャンクで 8b≈17秒 / 200字で≈34秒、＋毎回 TTFT・prefill。⇒ **当初方式は致命的**。
- **本設計（確定即表示＋LLMは追い上げレイヤ）**: 表示の主役は0遅延の確定テキスト。LLM整形は遅れても主役が崩れないため、**遅延はUX問題に格下げ**。⇒ **数秒遅延の実用リアルタイム議事録は成立**。
- **STT実測（DD-003 / faster-whisper・CPU・int8 / 本人音読70秒）**:
  - batch RTF: tiny 0.02 / base 0.05 / **medium 0.30**（<1）。**3秒チャンクstreamingは medium RTF 1.39〜1.78 で破綻**（30秒窓パディング）。**8〜12秒VAD区切りで medium streaming RTF 0.43〜0.61 に回復**（<1）→ §3.4「無音主体＋最大8〜12秒」を実証。
  - 日本語CER: **medium 0.26 ≪ base 0.54 ≪ tiny 0.82**（録音レベル低めで悲観側）。→ 会議中の確定表示にも **medium 級が必要**だが単体streaming RTF~0.5でCPU予算を食う。**二段STT（tiny/base即表示＋medium大窓 or 終了後再STT）**が現実解。
- **STT/LLM同時実行（DD-003実測）**: whisper(medium,4thr)併走で LLM 7.7→6.5 tok/s（-16%）。逆に Ollama(既定~8thr)併走で whisper RTF 0.61→**1.48（>1, スラッシング）**。**12スレッド/8コアの過剰割当が原因** → **既定は時間分離（§6）が必須**。同時に回すならスレッド総和≤物理コア（whisper=4 ＋ Ollama `num_thread`≤3）に厳格分割。
- **製品エンジン実測（DD-005 / pywhispercpp＝whisper.cpp）**: medium batch RTF 0.386 / **streaming@10s 0.688**（faster-whisper比 ~1.1〜1.4倍遅い、予想どおり）。**8〜12秒VAD・時間分離なら 0.688<1 を維持**（マージン約1.45倍＝薄め）。base streaming@10s は 0.098 と余裕 → 二段STTの即時側に使える。
- **VADは必須（速度でなく幻覚抑制・精度のため）**: whisper.cpp は無音/低レベルノイズで**幻覚**（偽の発話を大量生成）。**VADで有声だけ渡すと CER が約5倍改善**（faster-whisper 0.258→0.055 / whisper.cpp 0.567→**0.111**, DD-005）。残差は固有名詞/同音語（Tauri/要約/ケバ取り）＝**辞書置換で補正する対象**。→ §3.4 のVAD区切りは速度だけでなく**精度の前提**。簡易エネルギーVADでも実証、製品は Silero 等へ。
- ⚠️ 残課題: pywhispercpp のビルドが製品 whisper-rs と同SIMD最適化(AVX-512)か未確認＝倍率は目安。録音レベル低でCERは依然悲観側（適正録音で改善見込み）。実会議（被り/雑音/長尺）は未検証で 0.688 は楽観寄り。
