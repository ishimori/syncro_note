# 📂 議事録システム 要件定義書・仕様書（最終形 / 製品期）

本フォルダ（`doc/plan/要件/`）は、SynchroniNote の**最終形（製品期：Tauri + Rust + ローカルLLM）**の機能仕様を、開発者がそのまま実装に着手できる粒度でまとめた要件定義書・仕様書群です。**1機能 = 1ファイル**で分割しています。

---

## 位置づけ（重要）

- 📐 **設計SSOT**: 実装の基本設計（中核設計・タスク/データ構造・モデル選定・話者分離方式・モデル切替）は [基本設計書](../../spec/基本設計書.md) を**正**とします。本フォルダの File 1〜5 は**機能要件**で、基本設計書と食い違う場合は**基本設計書を優先**（特に File 2 のリアルタイム整形方式・File 3 の話者分離・File 5 のモデル名は再フレーム/訂正済み）。
- 本仕様は [企画書](../企画書.md) の最終形を、機能単位にブレイクダウンしたものです。背景・コンセプトは企画書を参照。
- **現在地は Phase 0（Python による評価フェーズ）** です。本仕様の機能群は [開発ロードマップ](../開発ロードマップ.md) の **Phase 2〜4** で段階的に実装されます。いきなり全機能を作る計画ではありません。
- 実装は各機能を [development-flow.md](../../development-flow.md) の5ステップで `/dd new` チケット化して進めます。

## 前提・未確定事項

- 文中の **モデル名（例: `Qwen3.6-7B` / `Qwen3.6-35B-A3B`）やメモリ前提（64GB）は最終形のイメージ値**です。実際に採用するモデル・量子化レベルは [開発ロードマップ §8（オープン論点）](../開発ロードマップ.md) と Phase 0 のベンチマーク結果で確定します。
- 技術スタックの「評価期（Python）↔ 製品期（Rust）」対応表は [開発ロードマップ §7](../開発ロードマップ.md) を参照。
- 動作環境の前提値（OS/CPU/RAM）と実機実測値の差分は [開発ロードマップ §2](../開発ロードマップ.md) を参照。

## 読み方・データの流れ

- 横断的な前提（システムアーキテクチャ・DBスキーマ）は **File 1** に集約しています。File 2〜5 は File 1 を前提に読んでください。
- リアルタイム中は **File 2（文字起こし）→ File 3（話者確定）→ File 4（人間メモ）** が同一タイムラインへ逐次蓄積され、会議終了後に **File 5** がそれらを統合して最終議事録を生成・永続化します。

```text
[会議中・リアルタイム]
  File 2 文字起こし ─┐
  File 3 話者確定   ─┼─▶ 統合タイムライン（時系列）
  File 4 人間メモ   ─┘            │
                                   ▼（会議終了トリガー）
[会議後・バッチ]            File 5 ディープ議事録生成 → SQLite 永続化
```
