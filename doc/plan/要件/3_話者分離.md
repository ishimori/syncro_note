## 【File 3】自動話者分離 ＆ 人間によるリアルタイム調教

> ⚠️ **設計SSOTに従属（根本訂正あり）**: [基本設計書](../../spec/基本設計書.md) §5 のとおり、**whisper.cpp に汎用の話者分離機能は無い**（`tinydiarize` は英語 small.en の「話者交代マーカー」のみ）。
> 本書 §2.1 の「whisper.cpp 組み込み diarization」前提は**成立しない**。話者IDは**外部スタック（Silero VAD ＋ ONNX話者埋め込み ＋ クラスタリング / pyannote / sherpa-onnx）**で生成し、会議中は仮ID＋人間確定（本書のT4フローは有効）。

本ドキュメントは、音声からの話者特徴の分離、ローカルLLM（Qwen）による文脈ベースの実名推測（名寄せ）、およびユーザー（人間）によるリアルタイムでの話者名一括修正ロジックに関する要件定義・仕様書である。

## 1. 話者分離・名寄せの全体論理構造

本システムでは、CPU環境での処理速度を維持するため、音声的な話者分離（重い処理）は最小限に留め、「LLMによる文脈からの実名推測」**と**「人間による一発修正」を組み合わせる3段構えのハイブリッドアプローチを採用する。

```text
1. 【音声分離 (Whisper)】 音声特徴から「Speaker_0」「Speaker_1」と機械的に識別
      ↓
2. 【AI名寄せ (Qwen)】  事前登録の参加者リストと会話内容から「Speaker_0 ＝ 鈴木」と推測
      ↓
3. 【人間調教 (Quasar)】 推測が間違っていた場合、ユーザーが「佐藤」に修正。
                        以降、バックエンド側でマッピングを固定・上書きする。

```

## 2. 各コンポーネントの仕様・要件

### 2.1 バックエンド：音声的な話者識別（Rust：`whisper-rs`）

* **仕様:**
    * 話者分離（Diarization）の手法は、`whisper.cpp` の実験的機能（`tinydiarize` は2話者前提）か、別途の軽量な話者識別ロジック／クレートから選定する（`whisper.cpp` 標準では汎用の多人数 Diarization は未対応のため、採用手法は評価フェーズで決定する）。
    * 音声チャンクごとに、誰が話したかを示すテンポラリなIDである `speaker_id`（例: `0`, `1`, `2`）を検出し、テキストセグメントと結合してLLMタスクおよびフロントエンドへ送信する。



### 2.2 バックエンド：Qwenによる「リアルタイム名寄せ」ロジック

* **仕様:**
    * リアルタイム整形タスク（File 2）において、Qwenに対し「会話の文脈からSpeaker_IDが誰であるかをプロファイリング（推測）せよ」という指示をシステムプロンプトに内包させる。
    * ただし、**「一度人間によって確定された話者マッピング」がある場合は、LLMの推測よりも人間の確定情報を絶対優先**して適用する。



#### 【バックエンドが保持する動的マッピング状態（メモリ構造案）】

Rust（TauriのState管理）またはフロントエンドのストアで、以下のState（状態）を常時保持・更新する。

```rust
// Rustでの状態管理イメージ
struct SpeakerMapping {
    // key: "Speaker_0", value: "鈴木（PM）"（確定した実名、未確定ならNone）
    map: HashMap<String, Option<String>>, 
}

```

## 3. Tauri IPC ＆ フロントエンド UI/UX 仕様

### 3.1 リアルタイムトークンの拡張（IPC通信）

File 2で定義した `realtime-token` イベントのペイロードに、`speaker_id` および現時点で判明している（またはAIが推測した）`speaker_name` を追加する。

* **イベント名:** `realtime-token`
* **ペイロード（JSON）:**
```json
{
  "chunk_id": "UUID",
  "speaker_id": "Speaker_0",       // 機械的に識別された話者ID
  "speaker_name": "Speaker_0",     // 人間未確定時はIDのまま（またはAIの推測名）
  "token": "お疲れ",
  "is_final": false
}

```



### 3.2 フロントエンドUI（Quasar）：話者の一括置換とセレクトボックス

* **UI構成:**
    * タイムライン（`q-chat-message`）の発言者名部分を、単なるテキストではなく、クリック可能なパーツ（`q-btn` または `q-select` を内包したチップ）にする。
* **人間の介入（調教）フロー:**
    1. 画面上に「Speaker_0」または間違った名前が表示されているのを見つける。
    2. ユーザーがその名前をクリックすると、**事前登録された参加者リストがドロップダウン（`q-select`）でポップアップ表示**される。
    3. ユーザーが「佐藤（エンジニア）」を選択する。
    4. フロントエンドは、TauriのCommand `update_speaker_mapping(speaker_id: "Speaker_0", real_name: "佐藤（エンジニア）")` を呼び出す。
    5. **【一括置換の実行】:** 画面上の過去のタイムライン、および現在進行形で生成されているテキストのうち、`speaker_id` が `"Speaker_0"` であるものの表示名を、**すべて瞬時に「佐藤（エンジニア）」へ一括置換**する。

## 4. プログラマーへの実装指示（File 3での実装範囲）

1. **ステップ1 (フロント):** Quasarで、チャット風タイムラインの発言者名をクリックした際、登録メンバーリストがドロップダウンで開き、名前を選択できるUIコンポーネントを作成する。
2. **ステップ2 (フロント):** オブジェクト/Map構造（例：`{"Speaker_0": "鈴木さん", "Speaker_1": "佐藤さん"}`）をVueの `computed` や `reactive` で管理し、話者名が書き換わったら過去の全発言の表示名が一瞬で連動して切り替わるVueのリアクティブロジックを組む。
3. **ステップ3 (バックエンド):** TauriのCommandとして `update_speaker_mapping` を実装し、フロントから送られてきた確定マッピング情報をRust側でも保持できるようにする。
4. **ステップ4 (統合):** 会議の途中で「Speaker_0 ➔ 鈴木さん」に手動修正した際、それ以降に新しく検知された「Speaker_0」の音声認識データが、自動的に「鈴木さん」としてフロントに降ってくる（LLMへのプロンプトにもその確定情報が反映される）連携を完成させる。
