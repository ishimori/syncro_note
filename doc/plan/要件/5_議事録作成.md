## 1. 終了後バッチ処理の全体シーケンス

会議中、CPUおよびメモリリソースを奪い合わないよう、会議終了のトリガーを契機に、バックエンド（Tauri）主導で「モデルのリロード（入れ替え）」を実行する。

```text
[ユーザー：会議終了ボタンをクリック]
       │
       ▼（Tauri Command発火）
1. 【軽量モデルの解放】  Ollama API経由（またはメモリ上から直接）Qwen 7B/9B をアンロード
       │
       ▼
2. 【特大モデルのロード】 Ollama API経由で `Qwen3.6-35B-A3B` (MoEモデル) をメモリに展開
       │
       ▼
3. 【コンテキスト統合】  「事前資料(Excel) ＋ 人間メモ ＋ 確定話者名付きタイムライン」を1つの巨大なプロンプトに結合
       │
       ▼
4. 【ディープ議事録生成】 Qwen3.6-35BがCPUフルスレッドでじっくり推論（Markdown形式で出力）
       │
       ▼
5. 【SQLite永続化】     生成されたマークダウン、終了時刻、ステータスをDBに保存 ➔ カレンダー画面へ遷移

```

## 2. 各コンポーネントの仕様・要件

### 2.1 バックエンド：モデルの動的切り替え（Tauri ➔ Ollama API）

CPU環境において、35BクラスのMoEモデルと7Bの密モデルを同時に起動することは、速度低下とメモリの無駄遣いに繋がる。そのため、Ollamaの「自動アンロード機能」または明示的なエンドポイント呼び出しを利用する。

* **仕様:**
    * 会議終了時、Tauriはまず `/api/generate` または `/api/chat` に対して、モデル名にリアルタイム用モデル（例：`qwen3.6:7b`）を指定し、`keep_alive: 0`（即時アンロード）をセットした空のリクエストを送るか、Ollamaサーバーのプロセス設定で制御し、メモリから完全に解放させる。
    * 続いて、清書用モデル **`qwen3.6:35b-a3b`** (GGUF量子化版) を指定して推論を開始する。これにより、メモリ64GBの大部分をこの最高精度モデルのコンテキストウィンドウに割り当てることができる。



### 2.2 プロンプトエンジニアリング：全文脈の統合

バックエンド（Rust）は、SQLiteおよびメモリバッファから以下の4つのデータを抽出し、1つの構造化されたプロンプトとして成形してQwenに投入する。

1. **会議の基本情報:** 会議名、アジェンダ、参加者リスト
2. **事前資料テキスト:** File 1でExcel等からパースしておいた `extracted_text`
3. **人間のリアルタイムメモ:** File 4で人間がタイムラインに差し込んだ手動補足
4. **文字起こしログ:** File 3で確定（実名化）させた全話者の発言タイムライン


## 3. Tauri IPC ＆ フロントエンド UI/UX 仕様

### 3.1 生成中のUI状態（Quasar）

* 清書処理が始まると、画面は「議事録生成中...（ローカルCPUで深く分析しています。1〜2分かかります）」というローカル処理特有のバッファリング画面（`q-inner-loading` や `q-spinner-gears`）に切り替える。
* Qwen35B MoEからの出力も、バッチ処理とはいえストリーミング（一文字ずつ表示）で画面の「最終プレビューエリア」に流し込み、進捗がユーザーに視覚的に伝わるようにする。

### 3.2 SQLiteへの永続化

Qwenから `is_final: true` を受け取り、すべてのテキスト（Markdown）が出力し終えたら、Tauriのバックエンド側でDBを更新する。

* **SQL実行クエリのイメージ:**
```sql
UPDATE meetings 
SET final_minutes = ?1, end_time = CURRENT_TIMESTAMP, status = 'completed' 
WHERE id = ?2;

```


* 保存完了後、フロントエンドはカレンダー画面（File 1）へ自動で遷移し、該当の日にちに「議事録完了」のバッジを表示する。

## 4. プログラマーへの実装指示（File 5での実装範囲）

1. **ステップ1 (バックエンド):** 会議終了のCommand `finish_meeting` を作成し、OllamaのAPIを叩いて `qwen3.6:35b-a3b` がCPU環境で正常にロードされ、長いテキスト（テスト用ダミーデータ）を食わせてもフリーズせずに回答を生成できるかを確認する。
2. **ステップ2 (バックエンド):** Rust側で、File 1〜4で集めたデータ（資料、メモ、タイムライン）を、上記「プロンプト構造案」の文字列に破綻なく結合（フォーマット）するビルダー関数を実装する。
3. **ステップ3 (フロント):** Quasarで「生成中」を表現するスピナーアニメーション画面を作り、Tauriから送られてくるマークダウン形式の議事録テキストが、`v-html` やマークダウンレンダラーライブラリ（`marked`など）を介して綺麗に画面にレンダリングされることを確認する。
4. **ステップ4 (統合テスト):** 会議開始から終了、DBへの保存、カレンダー画面へのリダイレクト、そしてカレンダーから過去の議事録（Markdown）がSQLiteから正しく読み出せるかの一連のワークフローを完遂させる。
