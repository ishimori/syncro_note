## 1. 全体共通仕様（システムアーキテクチャ）

本システムは、完全ローカル環境での低遅延動作とプライバシー保護を両立するため、以下の技術スタックを採用する。

### 1.1 開発環境・動作前提

* **OS:** クロスプラットフォーム（Windows / macOS / Linux）
* **実行環境:** **GPUなし（CPU駆動のみ）** / **物理メモリ: 64GB**
* **フロントエンド:** Quasar Framework (Vue 3 / Vite)
* **バックエンド:** Tauri v2 (Rust)
* **ローカルLLM/STTエンジン:**
    * Ollama（または `llama.cpp` のRustバインディング）
    * `whisper.cpp`（C/C++ ネイティブライブラリのRustラッパー）



### 1.2 技術的ハック（CPU環境への最適化）

メモリ64GBの潤沢なリソースを活かし、推論負荷（CPU）とメモリ消費のトレードオフを最適化する。会議中（リアルタイム）と会議終了後（バッチ）で、ロードするLLMモデルをバックエンド側で動的に切り替える（詳細はFile 5にて規定）。

## 2. 機能1：カレンダー画面・会議管理

ユーザーはアプリケーション起動後、まずカレンダーUIを起点として、過去の議事録の閲覧、および当日の会議の「事前登録」と「開始」を行う。

### 2.1 UI/UX 要件（フロントエンド：Quasar）

#### ① メイン画面（カレンダービュー）

* Quasarの公式拡張コンポーネントである **`@quasar/quasar-app-extension-qcalendar`** を使用して実装すること。
* 「月（Month）」「週（Week）」の切り替え表示に対応。
* SQLiteから取得した予定（タイトル、時間、ステータス）を各日付のセル内にレンダリングする。
* 過去の日付をクリックすると「議事録詳細モーダル」が開き、未来または当日の日付をクリックすると「会議作成/事前登録モーダル」が開く。

#### ② 会議作成・事前登録モーダル

会議を開始・予約する前に、以下のコンテキスト（前提知識）を入力するUIを設ける。

1. **基本情報:** 会議名、日時、場所/URL
2. **アジェンダ（議題）:** テキストエリア（Markdown対応）
3. **参加者リスト:**
    * チップス（`q-chip`）または入力フォーム形式。
    * 「氏名（必須）」「役職（任意）」「声の補足（例：男性、低音など・任意）」を入力可能とする。
4. **専門用語辞書:**
    * 社内用語、製品名、人名の特殊な漢字などをカンマ区切り、またはタグ形式で入力。
5. **参考資料ドロップゾーン:**
    * Excel（.xlsx）、PDF（.pdf）ファイルをドラッグ＆ドロップで受け付ける。



#### ③ 会議開始トリガー

* ステータスが「当日」の予定において、画面上に目立つ **「この予定で会議を開始する」ボタン（`q-btn`）** を配置。
* クリック時、登録されたすべての入力情報（コンテキスト）をシリアライズし、TauriのIPC（Command）を経由してRust側のセッションメモリに保持した上で、メインの文字起こし画面（File 2）へ画面遷移する。

## 3. バックエンド要件（Tauri / Rust）

### 3.1 データベース構造（SQLite）

ローカルでのデータ永続化のため、Rust側で `rusqlite` または `sqlx` を使用し、軽量な組み込みデータベース **SQLite** を構築する。データファイルはアプリのデータディレクトリ（`app_data_dir`）内に暗号化せずに保存する。

#### 【テーブル定義案】

```sql
-- 1. 会議マスター
CREATE TABLE meetings (
    id TEXT PRIMARY KEY,          -- UUID
    title TEXT NOT NULL,          -- 会議名
    agenda TEXT,                  -- アジェンダ
    start_time TIMESTAMP NOT NULL, -- 開始日時
    end_time TIMESTAMP,            -- 終了日時（終了時に更新）
    status TEXT NOT NULL,         -- 'scheduled', 'active', 'completed'
    final_minutes TEXT            -- 最終生成されたマークダウン議事録
);

-- 2. 参加者テーブル
CREATE TABLE participants (
    id TEXT PRIMARY KEY,
    meeting_id TEXT NOT NULL,
    name TEXT NOT NULL,           -- 氏名
    role TEXT,                    -- 役職
    voice_profile TEXT,           -- 声の特徴（話者分離のヒント用）
    FOREIGN KEY(meeting_id) REFERENCES meetings(id) ON DELETE CASCADE
);

-- 3. 専門用語辞書テーブル
CREATE TABLE vocabularies (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    meeting_id TEXT NOT NULL,
    word TEXT NOT NULL,           -- 単語
    FOREIGN KEY(meeting_id) REFERENCES meetings(id) ON DELETE CASCADE
);

-- 4. 添付ファイル管理テーブル
CREATE TABLE attachments (
    id TEXT PRIMARY KEY,
    meeting_id TEXT NOT NULL,
    file_name TEXT NOT NULL,      -- ファイル名
    local_path TEXT NOT NULL,     -- アプリ内にコピーしたローカル絶対パス
    file_type TEXT NOT NULL,      -- 'xlsx', 'pdf'
    extracted_text TEXT,          -- テキスト化（パース）した中身
    FOREIGN KEY(meeting_id) REFERENCES meetings(id) ON DELETE CASCADE
);

```

### 3.2 資料（Excel/PDF）のローカルパース処理

ユーザーから参考資料（Excel/PDF）が渡された場合、Rust側で即座にテキスト抽出処理を実行し、データベースにキャッシュする。

* **Excel処理:** `calamine` などのRustクレートを使用。シートごとにテーブル構造（Markdownのテーブル形式、またはJSONライクなテキスト）に変換して `extracted_text` に格納する。
* **PDF処理:** `pdf-extract` などのクレートを使用。プレーンテキストとして抽出して格納。
