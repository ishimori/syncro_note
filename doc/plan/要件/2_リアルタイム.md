本ドキュメントは、マイクからの音声入力、ローカル音声認識（Whisper）、およびローカルLLM（Qwen）によるリアルタイムテキスト整形とフロントエンドへのストリーミング表示を司るパイプラインの要件定義・仕様書である。

## 1. パイプライン全体のデータフロー

CPUへの負荷を最小限に抑えつつ低遅延を実現するため、バックエンド（Tauri/Rust）側で非同期タスク（Tokioタスク）を完全に分離し、以下の3フェーズのパイプラインを構築する。

```text
[マイク入力 (cpal)] ➔ (16kHzモノラル PCM) ➔ [音声バッファ]
                                                 │
  ┌──────────────────────────────────────────────┘ (VADによる発話区切り: 約2〜3秒)
  ▼
[Task 1: whisper.cpp (Base)] ➔ (生の文字起こしテキスト)
                                       │
  ┌────────────────────────────────────┘ (確定テキストの即時転送)
  ▼
[Task 2: Qwen (7B/9B)] ➔ (ケバ取り・リアルタイム整形)
                               │
                               └─ (1トークンごとに window.emit() 発火) ➔ [フロント (Quasar)]

```

## 2. 各コンポーネントの仕様・要件

### 2.1 音声キャプチャ（Rust：`cpal` クレート）

* **サンプリングレート:** 16,000Hz (16kHz) 固定。
* **チャンネル数:** 1チャンネル (モノラル) 固定。
* **フォーマット:** `f32` (32bit float PCM)。
* **仕様:**
    * 選択されたマイクデバイスから常に音声データをキャプチャし、Rustのリングバッファ（`ringbuf` クレート等）に流し込む。
    * **VAD（音声活動検知）:** 簡易的な音量エネルギー閾値、または軽量なVADロジックを実装し、「発話中」と「無音状態（約0.5秒〜1秒の無音）」を検知する。無音が検知されたタイミング、または最大3秒が経過した時点で、バッファ内の音声データを音声認識タスクへ引き渡す（音声チャンクの切り出し）。



### 2.2 音声認識（Rust：`whisper-rs` クレート）

* **推論エンジン:** `whisper.cpp` のRustバインディングである `whisper-rs` を使用。
* **採用モデル:** **`whisper-model-base.bin`**（または `tiny`）。
* **仕様:**
    * CPU環境での動作を最優先するため、`large` などの重いモデルは厳禁とする。
    * 切り出された音声チャンク（16kHz f32 PCM）を受け取り、即座にテキスト化を実行する。
    * 推論時は、Rustの `std::thread` または `tokio::task::spawn_blocking` を使用し、CPUのマルチスレッド設定（例：`params.set_n_threads(4)`）を適切に行うこと。
    * 得られた生のテキスト（例：「あー、えっと、次の、次のアジェンダですね」）を、即座にLLM処理タスクへフォワードする。



### 2.3 リアルタイムLLM整形（Ollama API 連携）

* **採用モデル:** **`Qwen3.6-7B-Instruct`**（または `Qwen3.6-9B-Instruct`）のGGUF（4bit量子化版）。
* **通信方式:** HTTP POST によるストリーミングリクエスト（`Ollama` の `/api/chat` エンドポイントを使用）。
* **仕様:**
    * Whisperから生テキストを受け取ったら、即座にOllamaへリクエストを飛ばす。
    * **プロンプト構造:** File 1でカレンダーから読み込んだ「参加者リスト」「アジェンダ」「専門用語辞書」をシステムプロンプトに動的に埋め込み、文脈の補正を行わせる。



#### 【リアルタイム整形プロンプト（システムプロンプト例）】

```text
# 前提知識
- 参加者: 鈴木（PM）、佐藤（エンジニア）
- 専門用語: Qwen, Tauri, SQLite

# 指示
あなたは優秀な会議の書記です。渡される音声認識テキストから「あー」「えっと」などのフィラー（ケバ）を完全に削除し、ビジネスにふさわしい綺麗な書き言葉にリアルタイムで整形してください。
前提知識にある正しい人名や専門用語への補正も行ってください。
余計な解説や挨拶は一切含めず、整形後のテキストだけを出力してください。

```

## 3. Tauri IPC ＆ フロントエンド通信仕様

LLMからの応答は1文字（1トークン）ずつストリーミングで返ってくるため、バックエンドはそれをバッファリングせず、即座にフロントエンド（Quasar）へイベント通知する。

### 3.1 Tauri イベント発行（Rust側）

Ollamaからストリーミングデータ（Chunk）を受信するたびに、以下のカスタムイベントを発火する。

* **イベント名:** `realtime-token`
* **ペイロード（JSON）:**
```json
{
  "chunk_id": "UUID",     // 発話チャンクごとの一意のID
  "token": "の",          // 1文字〜数文字のテキスト
  "is_final": false      // その発話チャンクの終了フラグ
}

```



### 3.2 フロントエンドでの受信と描画（Quasar / Vue 3）

* フロントエンドは `import { listen } from '@tauri-apps/api/event';` を使用して `realtime-token` を常時監視する。
* 同じ `chunk_id` を持つトークンを配列、または文字列として結合していき、Quasarの **`q-chat-message`** コンポーネント等を用いて画面上にリアルタイムにタイピングされているかのようにレンダリングする。
* `is_final: true` を受信した時点で、その発話ブロックのテキストを「確定」とし、タイムラインの配列（Vueの `reactive` 状態）に正式保存する。

## 4. プログラマーへの実装指示（File 2での実装範囲）

1. **ステップ1:** `cpal` クレートを用いて、マイクから 16kHz モノラルの音声データが正しく標準出力（またはログ）に出ることを確認する。
2. **ステップ2:** `whisper-rs` をプロジェクトに導入し、ローカルの `base.bin` モデルを読み込ませ、マイクに向かって話した言葉が数秒遅れで生のテキストとしてターミナルに表示される関数を実装する。
3. **ステップ3:** Ollamaで `qwen3.6:7b` を起動しておき、Rustの `reqwest` クレート等を使って、ストリーミング（`stream: true`）で応答を受け取り、1文字ずつ `println!` できることを確認する。
4. **ステップ4:** 上記すべてを統合し、Tauriの `window.emit` 経由で、Quasarの画面上のテキストエリアに文字がパラパラとリアルタイムに降ってくる画面を構築する。
