リアルタイムAI協調型ローカル議事録システム『SynchroniNote（シンクロニ・ノート、仮称）』の開発に向けた詳細企画書を作成しました。
グラボなし（CPU＋メインメモリ50GB）というハードウェア制約を最大限に活かし、Rustを核としたアーキテクチャで「低遅延・高セキュリティ・人間の思考を妨げないUI」を実現するための具体的な設計書です。長文ですが、実装のロードマップとしても機能するよう、極めて詳細に書き出しています。
------------------------------
## ソフトウェア開発企画書## 1. プロジェクト概要## 1.1 背景と課題
現在、GeminiやTeams、Zoomなどのクラウド型AI議事録ツールが普及している。しかしこれらには以下の2つの大きな課題が存在する。

   1. 事後処理の限界: 議事録が生成されるのは「会議終了後」であり、ミーティング進行中にAIの書き起こしを人間が修正したり、その場で重要事項をメモしてチームで共有・追記したりする「リアルタイムの協調」ができない。
   2. 機密情報の漏洩リスク: クラウド送信型のAIは、企業のインサイダー情報や顧客のプライバシーデータを扱う会議ではセキュリティポリシー上、使用できないケースが多い。

## 1.2 提案コンセプト
本プロジェクトでは、「完全ローカル（オフライン）環境」で動作し、「AIによるリアルタイム話者識別・書き起こし・要約」と、「人間によるリアルタイム追記・編集」を完全に同期させる、次世代のハイブリッド型議事録アプリケーションを開発する。
------------------------------
## 2. ターゲットスペック（動作環境）
本システムは、GPUを持たないが「大容量メインメモリ（50GB）」を搭載したWindowsノートPCで、CPU処理のみで実用的な速度（低遅延）を達成することを前提に最適化する。

* OS: Windows 11
* CPU: インテル Core i7/i9 相当、または AMD Ryzen 7/9 相当（マルチスレッド処理必須）
* GPU: なし（内蔵グラフィックス / CPU駆動）
* メインメモリ (RAM): 50GB（32GB以上をAIモデル駆動に、残りをシステムとUIに割り当て）

------------------------------
## 3. 主要機能要件## 3.1 リアルタイム話者識別＆声紋分析（Speaker Diarization）

* 機能: 1つの収音マイクから入力された音声波形から特徴量（アイデンティティ）を抽出し、「Aさん」「Bさん」「Cさん」と話者を識別する。
* 仕様: 会議開始時に各ユーザーの「声紋プロファイル（数秒の発話）」を登録するか、あるいはオンライン中に教師なし学習で自動クラスタリングを行い、話者をタイムスタンプごとに分離する。

## 3.2 高精度ローカル文字起こし（Speech-to-Text）

* 機能: 分離された話者ごとの音声セグメントを、リアルタイムで日本語テキストに変換する。
* 仕様: 音声入力を一定のチャンク（例: 2〜3秒単位）で区切り、バックグラウンドで並列パイプライン処理を行うことで、発話から文字化までのタイムラグを1秒未満に抑える。

## 3.3 コンテキスト追従型リアルタイム要約（LLM解析）

* 機能: 書き起こされたテキスト（生データ）を、日本語解析に強いLLMにストリーミングで流し込み、不要な「言い淀み（えーっと、あの、等）」を削除し、文脈を補正して議事録形式（箇条書き等）に整形する。
* 仕様: 1分〜3分ごとのトピックごとに区切り、会議の進行に合わせて「現在の要約」を常に自動更新し続ける。

## 3.4 人間×AI リアルタイム協調エディタ（Collaborative Editor）

* 機能: AIがリアルタイムに出力していく議事録エディタ上で、人間が同時にキーボードからメモを追記、修正、削除できる。
* 仕様: AIのストリーミング書き込みと、人間のタイピングが衝突（競合）してテキストが崩壊するのを防ぐため、分散同期アルゴリズムを採用する。

------------------------------

## 5. システム・アーキテクチャ＆技術スタック
GPUなし・大容量メモリ環境において、極限のパフォーマンスを発揮するためのハイブリッド構成。

                     +---------------------------------------+

                     |         UI層: Tauri (Frontend)        |
                     |       HTML5 / TypeScript / Yjs        |
                     +-------------------+-------------------+

                                         | IPC (Inter-Process)
                     +-------------------+-------------------+
                     |      コア層: Rust (Tauri Backend)      |
                     |  CPAL (マイク制御) / WebSocketサーバー   |
                     +---------+-------------------+---------+

                               |                   |
        +----------------------+                   +---------------------+

        | 外部プロセス: Python |                   | ローカルAPI: Ollama |
        | pyannote-audio       |                   | Qwen 3/3.5 (32B)    |
        | (声紋分析 / ONNX)    |                   | (CPU/AVX2最適化)    |
        +----------------------+                   +---------------------+

## 5.1 言語・フレームワーク選定

* コア＆アプリケーション基盤: Rust (Tauri)
* 理由: Electronのようにメモリを大食いせず、起動が極めて高速。マイクのオーディオバッファ（RAWデータ）を低遅延で処理するバックエンドをRustでネイティブ実装できる。
* UI層: TypeScript + React/Vue + Yjs
* 理由: エディタのUI構築はWeb技術が圧倒的に有利。CRDTライブラリであるYjs（またはAutomergeのRustバインディング）を使用し、AIの追記ストリームと、ユーザーのキーボード入力を同一ドキュメント内でデータ競合させずにマージする。

## 5.2 音声処理・AIパイプライン（GPUなし・メモリ50GB最適化）

   1. オーディオキャプチャ（Rust - cpal クレート）
   * WindowsのWASAPI経由で、収音マイクの音声をマルチスレッドで安全にキャプチャ。
   2. 声紋分析・話者識別（Python 外部プロセス or Rust ONNX runtime）
   * pyannote-audio などの事前学習済みモデルをONNX形式に変換し、Rustの onnxruntime クレートからCPUマルチスレッド（AVX2/AVX-512命令）で駆動。メモリ上に数GBの声紋ベクトルのインデックスを常駐させ、高速に話者を同定する。
   3. 文字起こし（Rust - whisper-rs）
   * OpenAIのWhisperをC/C++に移植した whisper.cpp のRustバインディング。CPU実行に完全に最適化されており、メモリ容量を活かして medium または large モデル（数GB）をメインメモリに丸ごとキャッシュし、高精度な文字起こしをCPUで行う。
   4. 日本語解析・要約（ローカルサーバー - Ollama）
   * バックグラウンドで Ollama をサービスとして常駐させる。
      * モデル選定: qwen3:32b（または3.5系） を採用。32Bモデル（約20GB〜24GB）は、50GBのメモリ環境であれば余裕を持ってOSのディスクキャッシュごとメモリ上に完全固定（mlock）できる。これにより、GPUがなくてもCPUの全コア（AVX2）を回すことで、実用十分なトークン生成速度（リアルタイム要約に追いつく速度）を確保する。

## 6. リアルタイム協調編集（CRDT）の実現メカニズム
人間とAIが同じテキストを編集する際の衝突を、以下のアルゴリズムで解決する。

   1. ドキュメントの抽象化: 議事録ドキュメントを、要素ごとの一意なIDを持ったツリー（または配列）構造のCRDT（Conflict-free Replicated Data Type）として定義する。
   2. AIの書き込みセクションの保護: AIが現在テキストを書き込んでいる最中の「最新の発言行」は、AIセクションとしてロック、または人間側エディタからは「追記のみ可能」な状態にする。
   3. 状態のイベント駆動同期:
   * Whisperがテキストを確定させる ➔ Rustバックエンドがイベントを発行 ➔ TauriのIPC通信経由でフロントのYjsドキュメントに「AIの発言（話者ID付き）」が非同期で差し込まれる。
      * 人間がタイピングする ➔ Yjsが即座にローカルドキュメントの状態をマージし、UIを更新する。

## 7. 開発ロードマップ（フェーズ別）## フェーズ 1: 音声・文字起こし基盤の構築（1〜2ヶ月）

* Rust（Tauri）プロジェクトの立ち上げ。
* cpal を用いたマイク音声入力の実装。
* whisper-rs を組み込み、CPU駆動でのリアルタイム文字起こし（チャンク処理）の精度と速度の検証。

## フェーズ 2: 声紋分析とOllama連携（1ヶ月）

* ONNXを用いたCPU駆動の話者識別（Diarization）モジュールの実装。音声を話者ごとに分離してWhisperに渡すパイプラインの完成。
* Ollama APIを経由し、qwen3:32b に対して「書き起こしテキストの整形・要約プロンプト」を送受信するストリーミング処理の実装。

## フェーズ 3: 協調エディタUIとCRDT実装（1.5ヶ月）

* Tauri（フロントエンド）での2ペイン画面の構築。
* Yjs の導入による、AIの自動追記と人間の自由入力が衝突しないマルチエントリエディタの実装。
* 左ペイン（AI）から右ペイン（人間）へのコンテキストコピー機能の実装。

## フェーズ 4: チューニングと実証実験（0.5ヶ月）

* 50GBのメモリ空間に対する、各AIモデルのメモリ割り当て最適化（スワップの発生防止）。
* CPU使用率のスパイクを抑えるための、スレッドプライオリティの調整。
* 実際の複数人ミーティングにおける、話者識別精度とリアルタイム要約のUX評価。

## 8. リスクと対策

* リスク1: CPU負荷の限界による音声のドロップ（音飛び）
* 対策: 音声のキャプチャ（最優先スレッド）、Whisper（中優先度）、Qwen（低優先度）と、Rust側でOSのスレッドスケジューリングの優先度を厳密に制御する。LLMの要約処理が重くなっても、マイクからの音声取りこぼしが絶対に起きない設計にする。
* リスク2: 複数人が同時に喋った際の話者識別精度の低下
* 対策: 初期バージョンでは「同時発話時は、メインの声（音量が大きい方）を優先」または「話者：同時発話」として分離するロジックを挟み、UXを崩さないようにする。
