## 1. 基本方針（企画書からの差分・補足）

企画書の完成形（リアルタイム協調・話者識別・Tauri/CRDT・CPUで32B）は妥当なゴールだが、4つの難所（CPUで32B実用速度／CPUでリアルタイムSTT／STTとLLMの同時実行／話者識別ONNX化＋CRDT）が同時に乗っており、初めてのローカルLLM開発でいきなり全部作るのはリスクが高い。そこで以下の方針を採る。

1. **評価フェーズはPython優先**
   企画書のコアは Rust(Tauri)/whisper-rs だが、「評価しながら進める」目的にはRustは回転が遅い。まず Python（uv/ruff）で価値とモデル性能を素早く検証し、**速度が問題になったホットパスだけ後からRustへ移植**する。Rust/Tauri/CRDTは協調UIを本気で作る段で投資する。

2. **バッチMVPから段階的にリアルタイム化**
   議事録ツールの中核価値（録音→文字起こし→要約→Markdown）はリアルタイム/話者識別/CRDTに依存しない。まずバッチCLIで中核を立ち上げて品質を評価し、その後にリアルタイム化・話者識別・協調UIを価値の高い順に積む。

3. **各フェーズに定量的な合否基準（評価指標）を置く**
   ローカルLLM/STTは「動く／動かない」だけでなく速度・精度の実測で判断する。指標は本書 §5 に定義し、各DDで測定する。

## 3. 全体アーキテクチャの段階的進化

```
[評価期] Python CLI
  音声ファイル → faster-whisper → (Ollama/Qwen 要約) → Markdown議事録

   ↓ ホットパス移植・リアルタイム要件が固まったら

[製品期] Tauri アプリ（企画書の最終形）
  cpal(Rust) → whisper-rs → diarization(ONNX) → Ollama → Yjs/CRDT 2ペインUI
```

## 5. 評価指標（各フェーズ共通）

| 指標 | 定義 | 目標の目安 |
|------|------|-----------|
| RTF (Real-Time Factor) | 音声長に対する処理時間の比。<1 でリアルタイム可 | リアルタイム化には RTF < 1（できれば <0.5） |
| CER / WER | 日本語文字起こしの誤り率（サンプル音声で測定） | 用途に耐える水準を実測で判断 |
| LLM tok/s | Ollamaの eval トークン生成速度 | 要約追従に必要な速度を P0-4 で見極め |
| E2E 遅延 | 発話 → 議事録反映までの体感遅延 | 企画書目標は「1秒未満」（CPUでは要検証） |
| メモリ占有 / スワップ | 同時稼働時のRAM使用とスワップ発生有無 | スワップ0を維持 |
| 要約品質ルーブリック | 決定事項/TODO/要点の網羅・正確さを3〜5段階で人手評価 | Phase1で基準化 |

## 7. 技術選定メモ（評価期と製品期）

| 領域 | 評価期（Python） | 製品期（企画書の最終形） |
|------|------------------|--------------------------|
| STT | faster-whisper (CTranslate2, CPU最適化) | whisper-rs (whisper.cpp) |
| LLM | Ollama + Qwen（8B/14B/32Bを比較） | 同左（モデルは実測で確定） |
| 話者識別 | pyannote-audio / 簡易手法 | ONNX Runtime (Rust) |
| 収音 | sounddevice 等 | cpal (Rust, WASAPI) |
| UI | CLI / 簡易表示 | Tauri + TypeScript + Yjs |
| Lint/Format | ruff | ruff (+ clippy/rustfmt) |

### 7.1 なぜ最終形は Tauri なのか（「TypeScript + FastAPI で良いのでは？」への回答）

> **結論**: Tauri の価値は「AI処理そのもの」ではなく、**「完全オフラインで完結する協調議事録アプリを、普通の人がインストールするだけで使える"製品"として配る」**ところにある。中身（録音→文字起こし→要約）は Python で詰め、固まってから最後に"製品の皮"として Tauri をかぶせる——という役割分担。「ギリギリまで Python」と「最後は Tauri」は矛盾しない。

**Tauri を選ぶ理由（この企画にとって効く順）**:

1. **2ペイン協調エディタを Web技術で作れる**: 最終形は「左＝AIの文字起こしが流れる／右＝人間がメモを書く」リアルタイムUI（§4 Phase 4）。この種の作り込みは Web(HTML/CSS/TS)の独壇場で、UI資産がそのまま使える。
2. **協調編集の核（Yjs/CRDT）が乗る**: 「同時に書いても壊れない」仕組み Yjs は Web/JS の道具。ここに自然に接続できる。
3. **インストールするだけで動く小さなアプリになる**: Windows用 `.exe`/`.msi` を生成でき、非エンジニアでもダブルクリック起動。サーバ常駐やコマンド操作が要らない。
4. **メモリが軽い（本機では決定的）**: Electron は Chrome を丸ごと抱えて重いが、Tauri は OS標準のWebView を借りるので軽量。**利用可能RAMが約9.6GBしかなく（§2）大半をローカルLLMに使いたい**本機では無視できない。
5. **（おまけ）速度が要る所だけ Rust 化できる**: 収音/STT等の低遅延部を後から cpal/whisper-rs へ。必要になってからでよい（§1-1）。

**「TypeScript + FastAPI で良いじゃん」と思ったとき、思い出すこと**:

TS+FastAPI（＝Python製サーバ＋ブラウザのWebアプリ）は評価期や暫定UIには十分良い選択。**だが最終形には合わない**。理由は——

- **"アプリ"でなく"起動中のサーバ＋ブラウザタブ"になる**: 使う人が FastAPI サーバを立ち上げ、ブラウザで開く運用。非エンジニアに配れない（単一のインストーラ成果物がない）。
- **メモリを二重に食う**: ブラウザ(Chrome) ＋ Python サーバが同時常駐 → 9.6GB の本機ではLLMの取り分を圧迫。Tauri(軽量WebView＋単一プロセス)が有利。
- **オフライン完結・ローカル保存と相性が悪い**: ブラウザ経由だとマイク権限・ファイル保存(ダウンロードダイアログ)・「クラウド送信なし」の保証が回りくどい。本企画のキモ（完全ローカル）は**ネイティブアプリの方が素直に守れる**。
- **どのみち UI と配布の課題は残る**: FastAPI で Python を続けても、フロントエンド実装とパッケージングは別途必要。「中身は Python で十分」は正しいが、それは Tauri を避ける理由にならない（Tauri が解くのは UI と配布であって、AI処理ではない）。

> 要するに **TS+FastAPI が解けるのは「中身の提供」、Tauri が解くのは「製品としての配布・UI・オフライン完結・省メモリ」** で、両者は競合しない。迷ったら「これはブラウザで開く社内ツールか？ インストールして使うオフライン製品か？」を自問する。本企画は後者。

## 8. 直近のオープン論点（要判断）

- **要約モデルの確定**: → **DD-002(P0-2)で暫定決着**。実測 tok/s（思考オフ・warmup1+3回平均）:
  qwen3:8b=8.9 / qwen3:14b=4.7 / gemma4:e4b=11.6 / **gemma4:26b(MoE活性4B)=12.6**。
  - **リアルタイム整形（採用するなら）= `qwen3:8b`**（TTFT 0.39秒最速・日本語固有名詞補正に強い。フォールバック e4b。14bは不可）。
  - **終了後バッチ清書 = `gemma4:26b`（MoE活性4B）**。「26B級品質を4B級速度で」を本機で実証。`Qwen3-30B-A3B` も比較候補。
  - 32B密モデルは未計測（gemma4:26b MoEで大型を代替評価）。
- **STT実機性能（→ DD-003 で決着）**: faster-whisper/CPU実測。**batch RTF: tiny 0.02 / base 0.05 / medium 0.30（<1）**。**3秒チャンクstreamingは medium RTF>1 で破綻 → 8〜12秒VAD区切りで 0.43〜0.61 に回復**。日本語CERは **medium 0.26 ≪ base 0.54 ≪ tiny 0.82**（要 medium）。**STT/LLM同時実行は無対策だと whisper RTF 0.61→1.48 でスラッシング → 時間分離が必須**。→ R3/R4 は「時間分離＋8〜12秒VAD＋二段STT」で対応方針確定。詳細は [基本設計書 §7](../spec/基本設計書.md) ・[DD-003結果](../DD/DD-003/結果.md)。
- **製品エンジン検証（→ DD-005 で決着）**: pywhispercpp(=whisper.cpp) 実測。medium streaming@10s RTF **0.688<1**（faster-whisper比 ~1.4倍遅いが、8〜12秒VAD・時間分離で成立）。**VADで無音を除くと CER が約5倍改善**（whisper.cpp medium **0.111**）＝高CERは無音幻覚が主因と判明。**製品エンジン基準でもリアルタイム可否ゲートPASS**（マージン薄め）。残るは実会議音声＋whisper-rsでの最終確認。詳細 [DD-005結果](../DD/DD-005/結果.md)。
- **話者分離（→ DD-004 系列で決着・採用確定）**: 評価ハーネスを整備（DD-010のsample02＝別人2名音読を流用し、正解RTTMを forced alignment で**自動生成**／DERは numpy 自前実装）。**採用＝`onnx-embed`（CAM++ ONNX 話者埋め込み）**（DD-004-1）。VAD→fbank→CMN→CAM++埋め込み→KMeans。モデル未配置時は baseline `simple-cluster`（MFCC+KMeans・無モデル・完全オフライン）へフォールバック。sample02 で onnx-embed 0.047／simple-cluster 0.036（無被り2話者の易しい素材で実質互角）。複数人・被り有り音声での精密評価と話者数自動推定は**据え置き**（素材準備待ち＝必要時に別DD）。製品期 Rust `ort` と地続き。詳細 [DD-004-1結果](../DD/DD-004-1/結果.md)。番号注: DD-004＝話者分離、DD-006＝モデル切替は基本設計書 §9 で予約済み（[本書 §4 Phase3](#)/[§6](#) 参照）。
  → **Whisper確定テキストの即表示を主役**／ケバ取りは辞書＋正規表現／**LLM整形は非ブロッキングの追い上げレイヤ**／深い構造化は終了後バッチ、へ再フレーム。設計SSOTとして [`doc/spec/基本設計書.md`](../spec/基本設計書.md) に確立済み。
