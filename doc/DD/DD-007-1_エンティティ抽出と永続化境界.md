# DD-007-1: エンティティ抽出と永続化境界の確定

| 作成日 | 更新日 | ステータス |
|--------|--------|------------|

> 親: [DD-007](DD-007_データベース基本設計.md)
> アプローチ: 標準（探索的・分析中心。SQLは書かず、抽出と境界線引きに集中）

## 目的

3情報源（SSOT／要件／モック）からエンティティを独立に抽出して突き合わせ、**SQLite に永続化する対象**を漏れなく確定する。SQL は書かない（DD-007-3/4 の入力を作る）。

## 背景・課題

- 単一情報源からの抽出は漏れる。[要件 File 1 の4表案](../plan/要件/1_共通仕様.md#L77-L118)には timeline と speaker mapping が無い。
- 会議中データの真実源は Rust メモリ（[SSOT §3.3 アクター](../spec/基本設計書.md#L140-L143)）。「メモリにある＝テーブル化」ではない。境界を引かないと過剰設計か耐久性欠落になる。

## 検討内容

### 1. 永続化境界（最重要）
| 区分 | 例 | DB化 |
|------|----|------|
| 永続化する | 会議メタ／参加者／用語／資料 extracted_text／確定タイムライン（原文）／人間メモ／確定話者マッピング／最終議事録 | ○ |
| 揮発（メモリのみ） | AudioChunk／RawSegment／LLM整形の中間トークン／クラスタリング途中状態 | ✕ |
| 別管理 | アプリ設定（マイク/モデル/スレッド/保存先, S-08） | ○（会議に属さない別表） |

### 2. 画面×データ項目マトリクス（[モック8枚](../mock/html/)）
S-01〜S-08 で「入力／表示／永続化先」を1行ずつ列挙し、書き込み元の無い表示項目を検出する（成果物は `DD-007-1/画面項目マトリクス.md` に分離）。

### 3. ライフサイクル追跡
`scheduled → active →（generating）→ completed` ＋ **強制終了からの復旧**（[File 4 §3.1 暫定ログ](../plan/要件/4_メモ追記.md#L62-L72)）。各遷移の read/write を追い、status enum と耐久性の漏れを検出。

### 4. 突き合わせで判明した不足エンティティ（暫定）
- `timeline_elements`（中核・要件案に無い）: ai_transcription / human_memo。**原文 immutable と LLM整形は別フィールド**。
- `speaker_mappings`（中核・要件案に無い）: 会議×speaker_id で confirmed / ai_guess の2層。
- `app_settings`（S-08）。※スキーマ版管理テーブルは作らない（移行不要・起動時 schema.sql 適用のみ／DD-007-4）。

## 決定事項

成果物（添付）に集約。要点:
- **永続化境界**を3分類で確定（[永続化境界.md](DD-007-1/永続化境界.md)）。会議中の真実源はメモリ、SQLiteへは batch-flush 対象のみ。**暫定ログ表は作らず `timeline_elements` の逐次flushで耐久性を兼ねる**。
- **画面×項目マトリクス**で孤児表示を0に解消（[画面項目マトリクス.md](DD-007-1/画面項目マトリクス.md)）。
- **確定エンティティ7表**＝要件4表（拡張）＋新規 `timeline_elements`・`speaker_mappings`・`app_settings`。status enum に `generating`/`aborted` 追加。`voice_profile`→`voice_hint`。新規カラム `batch_model`/`generation_seconds`/`audio_path` 等（[状態遷移とエンティティ確定.md](DD-007-1/状態遷移とエンティティ確定.md)）。
- SSOT §8 訂正一覧の全行を反映先にマッピング済み（未反映0）。
