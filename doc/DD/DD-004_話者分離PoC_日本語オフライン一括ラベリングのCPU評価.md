## 目的

「whisper 非依存の話者分離（diarization）」を **CPU・完全オフライン**で実用に載せられるか、手法を実測比較して**評価期に採用する1手法を決める**。具体的には日本語 2〜4 話者の音声を、終了後バッチで「話者A/B/C…」に**一括ラベリング**する安定度・精度・速度・導入難度を測る。

- 対応ロードマップ: [Phase 3 / P3-1 手法評価](../plan/開発ロードマップ.md#L107-L109)
- 対応SSOT: [基本設計書 §9 検証計画](../spec/基本設計書.md#L238)（DoD=「日本語2〜4話者のオフライン一括ラベリング安定度」）

## 背景・課題

- 要件 File 2/3/5 は「whisper.cpp に話者分離がある」前提だが**誤り**。汎用話者分離は別スタック（pyannote.audio / sherpa-onnx / ONNX埋め込み＋クラスタリング）が必要 — [基本設計書 §6.x](../spec/基本設計書.md) で訂正済み。
- [DD-008結果](DD-008/結果.md#L39) で、現行MVPの話者帰属は**話者分離なしのLLM推測**で誤帰属リスクが顕在化（qwen3:8b版で不安定）。→ 本PoCの成果が帰属精度の改善元になる。
- 設計方針（[基本設計書 §6](../spec/基本設計書.md#L189-L191)）は **会議中はリアルタイム埋め込み＋クラスタリング**、**終了後はオフライン一括ラベリングで精度を取る**の二段構え。本DDは後者（終了後バッチ）を先に評価する（負荷が軽くマージンを測りやすいため）。

## 検討内容

評価期 Python（CPU）で比較する候補と観点。

| 候補手法 | 概要 | 期待 | 懸念 |
|---------|------|------|------|
| **pyannote.audio** | 定番。segmentation + embedding + clustering の一体パイプライン | 精度の基準値。導入容易 | HF token / モデルDL要、CPUでの速度、ライセンス確認 |
| **sherpa-onnx** | ONNXランタイム前提。製品期(Rust `ort`)と地続き | 製品期移植が楽、軽量 | 日本語実績・パイプライン組み立ての手数 |
| **簡易クラスタリング自作** | Silero VAD → ECAPA/3D-Speaker 埋め込み(ONNX) → 凝集型/スペクトラルクラスタリング | 依存最小・透明性 | 話者数推定・閾値調整が自前、精度未知 |

**評価指標**（§5 に追加想定。本DDで測る）:

| 指標 | 定義 | 合否の目安 |
|------|------|-----------|
| DER (Diarization Error Rate) | miss + false alarm + speaker confusion の割合 | 用途に耐える水準を実測で判断（まず基準値取得） |
| 話者数推定精度 | 正解話者数と推定数の一致率 | 2〜4話者で取り違えが許容内か |
| RTF | 音声長に対する diarization 処理時間比 | 終了後バッチなので RTF<1 は必須ではないが実用時間内か |
| 導入難度 | モデル入手・依存・ライセンス・実装手数 | 評価期/製品期それぞれで現実的か |
| STT統合の容易さ | 分離区間を whisper へ渡す統合の手数（P3-2 への布石） | 区間→話者ラベル付き文字起こしが組めるか |

## 決定事項

- **一次評価素材 = [`python/audio/sample02.wav`](../../../python/audio/sample02.wav)**（DD-010 の評価クリップを流用）。モノラル/16kHz/120秒、**別人2名**（鈴木＝進行役／佐藤＝報告役）が交互音読した会議風対話。話者ラベル・発話順は [DD-010 台本](DD-010/台本.md) で確定済み。
- **参照RTTMは手作業アノテ不要で自動生成**する方針: `script02.txt` を音声に forced alignment → 各セグメントを台本のターン順に突合し話者ラベル（鈴木/佐藤）を付与 → RTTM 化。交互・順序確定・オーバーラップ無しの音読構成だから成立する。これにより Phase 1 のブロッカー（正解RTTM不在）を手動アノテーションなしで解消（DAリスク#1）。
- **ただし sample02 は diarization的には“易しいケース”**（オーバーラップ無し・2話者固定・クリーン単一マイク）。一次ベースライン／パイプライン疎通に使い、**DERは楽観値**と解釈する。**オーバーラップあり・3〜4話者の追加素材は Phase 2 の残課題**として別途用意し、これ単体で採用判断しない。
- 採用手法・暫定パラメータ・評価期↔製品期の移植方針は PoC 実施後にここへ追記する（現時点で手法は未確定）。

### Phase 0 DA批判レビュー

**DA観点:** PoCの前提（評価データ・指標の妥当性）が崩れると、後続の手法比較すべてが無意味になる。

| # | 発見した問題/改善点 | 重要度 | 再現手順（高/中は必須） | DA観点 | 対応 |
|---|-------------------|--------|----------------------|--------|------|
| 1 | 正解RTTMが無いと DER を測れず、手法比較が主観評価に退化する | 高 | Phase2でDER列が空のまま採用判断に進む → 根拠なき決定 | 前提条件の検証 | ✅ sample02（別人2名・交互音読）＋`script02.txt` の forced alignment で参照RTTMを自動生成（手動アノテ不要）。決定事項に反映 |
| 2 | pyannote のオフライン/ライセンス制約: 学習済みパイプラインは **HFでgated＋初回DLにネット必須**。「完全オフライン・社外送信禁止」要件と衝突しうる | 中 | 初回 `from_pretrained` で HF token要求／ネット接続発生 | 制約整合性 | ⏭️ Phase 2: ①初回のみ事前DL→ローカルキャッシュで運用可否を確認 ②再配布制約を確認。**抵触なら sherpa-onnx(Apache)／簡易自作(ONNX埋め込み)を優先**（製品期 Rust `ort` とも地続き）。`pyannote.metrics` 自体は gateなし・モデル不要で計測には支障なし |
| 3 | sample02 は易しいケース（無オーバーラップ・2話者固定）で DER が楽観的に出る → これ単体採用は製品で滑る | 高 | 実会議のクロストークで speaker confusion 多発 | 前提の代表性 | ⏭️ Phase 2 までにオーバーラップあり・3〜4話者素材を1本追加。sample02は一次ベースライン限定 |
| 4 | **参照RTTMの境界精度が DER の下限を縛る**: whisper word timestamps の誤差ぶん、完璧な diarizer でも DER が 0 にならない | 中 | 境界±数百msの誤差がそのまま confusion/miss に計上 | 計測手法の妥当性 | ✅ 境界は話者交代の**無音ギャップ中点**に置く＋DER は **collar=0.25s** で吸収（[Phase1詳細設計 §2,§4](DD-004/Phase1詳細設計.md)）。相対比較には十分 |
