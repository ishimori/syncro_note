> アプローチ: 標準（探索的実装＋実測スパイク）。画面は持たず **ターミナル/ログで検証**。番号: [基本設計書 §9](../spec/基本設計書.md) で予約済みの DD-006 を使用（004=話者分離PoC と共に欠番確保されていた）。

## 目的

会議終了時の **「ライブ整形モデル(qwen3:8b) → 終了後バッチ清書モデル(gemma4:26b)」のモデル切替シーケンス**を評価期Pythonで実測し、(1) **二重常駐窓が出ない**（両モデル同時ロードによる OOM/スワップを起こさない）こと、(2) **実切替秒数と RSS 推移**、(3) **清書の実所要時間**、を数値で確定する。成果は [基本設計書 §4.3/§6](../spec/基本設計書.md) の切替手順の裏付けと、**UI文言（「清書中… 約N分」）の確定**。

## 背景・課題

- [基本設計書 §4.3（モデル切替）](../spec/基本設計書.md)・[§6（終了後バッチ清書）](../spec/基本設計書.md)・[§9](../spec/基本設計書.md) で切替方針（`OLLAMA_MAX_LOADED_MODELS=1` ＋ `keep_alive:0` で退避 → `/api/ps` 空確認 → 特大MoEロード）は決めたが **未実測**。§9 で DD-006 として検証予約。
- 実機の **利用可能RAMは実測変動が大きい**（[ロードマップ §2](../plan/開発ロードマップ.md): 約9.6GB の記録／**Phase 0 実測時は 43.4GB 空き**）。低RAM時に qwen3:8b＋gemma4:26b が **二重常駐すると OOM/スワップ窓**（合計≈21.6GB）が開く。切替時の「空き確認」実挙動（特に **Windows の早期/遅延 unload**）を確かめる必要がある。
- 既知の落とし穴: ①client の stream 中断で **Ollama サーバ側生成が即停止する保証がない**（[§3.4](../spec/基本設計書.md)）。②`keep_alive` は既定5分・Windows 挙動に振り回される（[§4.3](../spec/基本設計書.md)）。③現状 `OLLAMA_*` env 未設定＝`MAX_LOADED_MODELS` デフォルト（サーバ起動済みのため env 反映には再起動要）。これらが切替総時間・二重常駐窓に効く。
- UI: [§6](../spec/基本設計書.md) は清書を **実3〜6分**と見積もり、「1〜2分」は過小と指摘。**実測してUI表示文言を確定**する（DoD）。

## 検討内容

- **検証シーケンス**（[§3.4 finish手順](../spec/基本設計書.md) の評価期Python版。Ollama は SDK アクセスなので言語非依存＝Python実測が製品Rustへ転用可）:
  1. **live**: `qwen3:8b` を `keep_alive=-1` でロード（短い整形プロンプトで warmup・常駐確認）
  2. **switch**: `qwen3:8b` に `keep_alive=0` を投げて退避要求
  3. **wait**: `ollama.ps()` をポーリングし 8b が消えた（空 or 8b不在）ことを確認
  4. **batch**: `gemma4:26b` をロードし **本物の清書プロンプト**（`summarize.build_minutes_prompt`）で生成
- **測定指標**:
  - **切替総秒数**＝step2要求〜step4のbatch TTFT。内訳を **unload待ち** と **26bロード+TTFT** に分解
  - **`ollama.ps()` 時系列** → **二重常駐窓の有無**（8b と 26b が同時に載る瞬間があるか）
  - **RSS 時系列**（`psutil`：ollama server ＋ runner子プロセス合算）→ ピークRAM・スワップ有無
  - **清書実時間**＝wall-clock ／ eval tok/s × 出力tok（[§5 指標](../plan/開発ロードマップ.md)）
- **実機状態（Phase 0 実測 2026-06-07）**: pull済み `qwen3:8b`=4.87GB / `gemma4:26b`=16.75GB(25.8B) / `gemma4:e4b` / `qwen3:14b`。ロード中=なし。空き43.4GB。`OLLAMA_*` env なし。→ **潤沢メモリ時は二重常駐してもOOMしない**ため、DoD「二重常駐窓なし」は *低RAM時の安全性を潤沢な今に確認* する位置づけ。窓検知は `ollama.ps()` で秒単位に行いメモリ余裕と独立に判定する。
- **再利用 / 新規**（Phase 0 実読で確定）:
  - 再利用 = `bench/llm_bench.py`（`run_once`＝streamでTTFT/eval_count/eval_duration計測・`tokens_per_second`・`BenchResult`）＋ `pipeline/summarize.py`（`build_minutes_prompt`／`stream_summarize`＝本物の清書プロンプトとメトリクス）。Ollamaは **`ollama` Python SDK** 経由（直叩きしない）。
  - 新規 = `ollama.ps()` ポーリング（空確認・二重常駐窓検知）、`psutil`（dev依存追加）での **server＋runner子合算RSS**、切替シーケンス制御。

## 決定事項

- 評価期Pythonの単発ベンチ `python/src/synchroni_note/bench/model_switch_bench.py`（新規）で実測。**CLIエントリ（pyproject）は作らない**（`python -m ...bench.model_switch_bench` で実行。pyproject を触らず別セッションと衝突回避）。
- 切替方式 = [§4.3](../spec/基本設計書.md) の「`keep_alive=0` → `ollama.ps()` 空確認 → ロード」を **スクリプト側で順序保証**して計測（env無しでも二重常駐を防げるかを検証。`MAX_LOADED_MODELS=1` 比較は任意）。
- **合否ゲート（[§9](../spec/基本設計書.md)）** = **二重常駐窓が出ない** ＋ **清書実時間を実測してUI文言を確定**。
- モデルは Phase 0 で実在確認済み（`qwen3:8b` / `gemma4:26b`）。RSS は `psutil` で server＋runner子合算。
- 出力 = `DD-006/結果.md`（切替秒数の内訳・`ollama.ps()` 時系列・RSSピーク・清書実時間・**UI推奨文言**）。
