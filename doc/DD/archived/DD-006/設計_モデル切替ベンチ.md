## データフロー（計測点つき）

1. **preflight**: `ollama.ps()` が空か確認（非空なら warn＋既存を `keep_alive=0` 退避）。RSS baseline 取得。
2. **live ロード**: `run_once("qwen3:8b", 短い整形prompt, keep_alive=-1)`（warmup）。`ollama.ps()` で 8b 常駐確認・RSS記録。→ `t_live_ready`
3. **switch要求**: `ollama.generate("qwen3:8b", "", keep_alive=0)`（空生成でアンロード要求）。→ `t_switch_req`
4. **unload待ち**: `ollama.ps()` を `POLL_S`(0.2s) 間隔でポーリングし 8b が消えるまで。各回 ps名一覧＋合算RSSを時系列 append。**二重常駐窓検知**: いずれかで `{qwen3:8b, gemma4:26b} ⊆ names` なら `double_resident=True`。→ `t_unloaded`（`UNLOAD_TIMEOUT_S`=30s）
5. **batch清書**: `build_minutes_prompt(transcript)` → `run_once("gemma4:26b", prompt, keep_alive=-1)`（streamで TTFT＝`t_batch_ttft`、eval_count/eval_duration＝清書速度）。生成完了 `t_done`。26b のみ載っていることを確認。
6. **集計**:
   - `switch_total_s = t_batch_ttft - t_switch_req`（＝UIの「切替待ち」体感）
   - `unload_wait_s  = t_unloaded   - t_switch_req`
   - `batch_load_ttft_s = t_batch_ttft - t_unloaded`
   - `cleanup_s = t_done - t_batch_ttft`（清書本体）／清書 tok/s
   - `double_resident: bool`, `rss_peak_gb`, スワップ有無（任意）
7. **出力**: コンソール表＋`--out ../doc/DD/DD-006/結果.md`。cold（初回ロード）/ warm を区別して記録。

## 関数構成（案）

- `sample_ollama_rss() -> float`: `psutil` で `ollama` 系プロセス（server＋`children(recursive)`）の RSS 合計GB。
- `wait_until_unloaded(target, others) -> (wait_s, double_resident, ps_timeline)`: ps空ポーリング。
- `run_switch_cycle(live, batch, transcript, *, cold) -> SwitchResult`: 上記 2〜6 を1サイクル。
- `SwitchResult` dataclass: 各秒数＋`double_resident`＋`rss_peak_gb`＋清書tok/s。
- `format_markdown(results)` / `main(argv)`。

## 👀 レビュー論点（ユーザー判断が要る点）

1. **RSS手段**: `psutil`（dev依存追加）で runner子合算 ←推奨。代替＝`ollama.ps()` の `size` のみ（プロセス実RSS/スワップは取れない）。どちらにするか。
2. **アンロード方法**: 空プロンプト＋`keep_alive=0` の `ollama.generate` で退避 ← これでよいか（ollama SDK の通例）。
3. **清書入力 transcript**: 清書実時間（狙い 3〜6分）を測るには相応の長さが要る。候補＝`bench/sample_transcript.txt`（DD-002流用・長さ未確認）／DD-010 `script02.txt`(435字＝短い)／**より長い実会議相当を用意**。短いと「3〜6分」を再現できずUI文言の根拠が弱い。**どれを使うか**。
4. **MAX_LOADED_MODELS**: まず env無し（現状）で「ps空待ち方式」が二重常駐を防げるか測る ←推奨。`OLLAMA_MAX_LOADED_MODELS=1`（要サーバ再起動）での比較も行うか。
5. **ポーリング**: `POLL_S=0.2s` / `UNLOAD_TIMEOUT_S=30s` でよいか。

## エッジ / エラー

- Ollama 未起動 → 即 fail（明確なメッセージ）。
- unload タイムアウト → 記録して中断（「二重常駐リスクあり」と明示）。
- 生成 stream の途中中断は本ベンチでは扱わない（[§3.4](../../spec/基本設計書.md) の「中断が即停止しない」issue は別検証）。
