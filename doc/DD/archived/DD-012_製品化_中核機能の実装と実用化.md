## 目的

DD-011 で「Tauriの皮 ↔ Python の中身」をつなぐ最小経路（サンプル音声→S-05へ逐次表示）を通した。本DD（親）はその先、**「会議する→議事録が残る→見返す→配れる」という製品の中核ループを完成させ、実用に耐える状態にする**ことをゴールとする。実装は下記の子DDで分割して行い、本DDは全体像・順序・依存の管理に徹する。

## 背景・課題

- 中核価値（録音→文字起こし→要約→Markdown）は Python で実証済み（DD-008/DD-010）。設計はSSOT（基本設計書／DD-009画面設計／DD-007 DB）で凍結済み。皮（Tauri/Quasar）も DD-011 で「起動・S-05骨格・全画面モック＋ルーター」まで到達。
- 残るは「皮と中身を本接続して**実際に使える**ようにする」こと。現状は各画面がモック表示で、マイク・保存・要約・配布が未接続。
- 協調UI(CRDT)・Rust移植・実会議評価（ロードマップ Phase 4-2/4-3・Phase 5）は最もリスクが高く、計画上も最後。本DDは**そこへ行く前の「単独でも実用になる製品」**までを射程にする。

## 進め方（順序と並行）

1. **DD-012-1（マイク）を最初に** — DD-011 **3-C（ファイル入力での配線）完了後**、その土台にマイク差分を足すだけで「実会議で使える」へ一気に近づく＝費用対効果が最大。
2. **DD-012-2 と DD-012-3 はペア**（議事録が残る一周）。触る領域が別（清書パイプライン vs DB層）なので**複数セッションで並行可**。012-2 の「保存」は最初ファイル、012-3 完了後に DB へ差し替え。
3. DD-012-4／-5 で体験を厚く（-5 は **DD-004-1 の採用確定が前提**）。最後に DD-012-6 で配布可能に。

## スコープ外（将来・別系統で起票）

- 同時編集 Yjs/CRDT（ロードマップ **P4-2**）／ホットパスの Rust 移植（**P4-3**, cpal/whisper-rs）／メモリ・スレッド最適化・実会議UX評価（**P5**）。最もリスクが高く最後に投資するため本DDには含めない。
- 文字化け・逐次バッファリング・子プロセス kill の基本対処は **DD-011 Phase3 に内包済み**（確実版 kill のみ 012-6 で仕上げる）。

- ✅ 子DD **012-1〜012-3** が完了し、実ウィンドウで「**マイクで会議→終了→議事録が清書・保存→カレンダーで見返す**」が一周する。
- ✅ 体験向上（012-4／-5）含め、子DD **012-1〜5・7〜14** をすべて完了・アーカイブ。中核ループは単独で実用可能。
