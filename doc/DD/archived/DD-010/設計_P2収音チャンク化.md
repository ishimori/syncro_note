# DD-010 Phase 2 実装前詳細化 — 収音＋VADチャンク化（P2-1）

> 📐 コーディング前の設計合意用。👀 レビュー後に着手する。
> 関連: [基本設計書 §3.2/§3.4](../../spec/基本設計書.md)（チャネル/背圧）・[DD-003結果](../DD-003/結果.md)（8〜12秒VAD区切り）・再利用元 `bench/vad_segment.py`。

## 1. 課題

`vad_segment.detect_voiced_spans()` は**全体配列前提（バッチ）**。ライブのマイク入力では音声が少しずつ届くため、**ローリングに区切る状態機械**が要る。加えて収音は専用スレッド（`sd.InputStream` コールバック）で走るため、**スレッド境界と背圧**を設計する。

## 2. 全体データフロー

```
[音源] ──blocks──▶ [入力キュー] ──▶ [VadChunker] ──Chunk──▶ [チャンクキュー(bounded)] ──▶ (Phase3 STT)
  │                  (queue.Queue)      状態機械            maxsize=4 / 満杯はdrop+計数
  ├ mic : sd.InputStream コールバック（別スレッド・即push・重い処理禁止）
  └ file: sample02.wav を mic 代替で流す（再現測定用・--realtime で実時間ペース）
```

- **音源を抽象化**: `mic`（ライブ）と `file`（録音クリップを同一経路で流す）の2モード。**測定は file モードで再現性を確保**、体感は mic / `--realtime` file で確認。
- コールバックは**即 push のみ**（VAD/STTを呼ばない＝オーバーラン防止。[基本設計書 §3.2](../../spec/基本設計書.md) の「コールバックは軽く」に対応）。

## 3. チャンク化方式（状態機械）

DD-003 の「**無音主体・最大8〜12秒**」を踏襲。ストリーミングでは全体中央値が取れないため**固定の絶対エネルギー閾値**で無音判定する（`vad_segment` の `abs_floor=0.004` を流用。適応閾値は将来課題）。

- 蓄積バッファ `buf` に新ブロックを連結し、**フレームRMS**（`vad_segment.frame_rms` を再利用）で各30msフレームの有声/無音を判定。
- **チャンク確定（emit）条件**（いずれか）:
  1. `buf` の有声長 ≥ `min_speech_s`（既定2.0s）**かつ** 末尾に `silence_ms`（既定400ms）以上の無音 → **無音の中点でカット**（単語切れを避ける）。無音以降は次チャンクへ繰り越し。
  2. `buf` 長 ≥ `max_seg_s`（既定10s, 8〜12sレンジ）→ **強制カット**（繰り越しあり）。←連続発話での単語切れリスク。Phase 3 で CER差分として測定。
- 末尾の端数は `flush()` で最終チャンクに。

## 4. I/F（提案）

```python

import numpy as np

SAMPLE_RATE = 16000

@dataclass
class Chunk:

    samples: np.ndarray      # float32 / 16k / mono
    t_start_ms: int          # capture基点からの相対ms（体感遅延の計測用）
    t_end_ms: int

    """ローリングVADで Chunk を切り出す状態機械（純ロジック＝pytest対象）。"""
    def __init__(self, *, sr=SAMPLE_RATE, frame_ms=30, abs_floor=0.004,
                 silence_ms=400, min_speech_s=2.0, max_seg_s=10.0): ...
    def push(self, block: np.ndarray) -> list[Chunk]:
        """新ブロックを蓄積し、確定したチャンク（0個以上）を返す。"""
    def flush(self) -> list[Chunk]:
        """ストリーム終端で残バッファを最終チャンク化。"""

def capture_mic(chunker: VadChunker, out_q, *, device=None, block_ms=100) -> None:
    """sd.InputStream(16k/mono/f32) でライブ収音し、確定チャンクを out_q へ。"""

def capture_file(path, chunker: VadChunker, out_q, *, realtime=False) -> None:
    """録音WAVを mic 代替で流す（再現測定用）。realtime=True で実時間ペース。"""
```

- `out_q = queue.Queue(maxsize=4)`（bounded）。満杯時は **put_nowait の失敗を捕捉して drop計数＋警告ログ**（[基本設計書 §3.4](../../spec/基本設計書.md) のドロップ計数のPython簡易版）。STT単体・RTF~0.5では基本溢れない前提の安全弁。
- `VadChunker` は I/O を持たない純ロジックなので**合成波形で単体テスト可能**（`vad_segment` と同方針）。

## 5. エッジケース・エラーハンドリング

| ケース | 方針 |
|--------|------|
| デバイスが16k非対応 | `sd.InputStream(samplerate=16000)` を試行→失敗時はデバイス既定SRで収音し**16kへリサンプル**（または明確にエラー表示）。file モードは `faster_whisper.audio.decode_audio` が16k化。 |
| 全区間無音 | チャンク0個（正常）。`flush()` も空。 |
| 連続発話 > max_seg_s | 強制カット（単語切れ可能性）。Phase 3 の CER差分で影響を定量化、必要なら窓オーバーラップを後続検討。 |
| チャンクキュー満杯 | `drop計数++`＋ログ。音声は捨てる（STTが追いつかない兆候としてメトリクス化）。 |
| 終端の端数 | `flush()` で最終チャンク化（取りこぼし防止）。 |

## 6. テスト観点（pytest）

- `VadChunker.push`: 「有声2s→無音500ms→有声」で**無音中点カットの2チャンク**になる。
- `max_seg_s` 超の連続有声 → 強制分割される。
- 無音のみ → 空。
- `capture_file(sample02.wav)`: チャンク列が音声長を概ね被覆（合計有声秒 ≈ VADバッチ結果）。
- 🔬 機械検証（Phase 2 DoD）: 録音クリップを `capture_file` で流し、**8〜12秒チャンク列が安定生成**されるログ。
