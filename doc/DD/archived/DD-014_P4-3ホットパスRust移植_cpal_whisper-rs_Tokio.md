## 目的

Python 評価期（DD-010）で確立した「**収音→STT→LLM連携**」のホットパスを Rust(Tauri backend)へ移植し、低遅延・低CPU・音飛びゼロを実現する。`cpal 収音 → ringbuf → Silero VAD → whisper-rs → reqwest(Ollama stream)` を Tokio 4タスク（T_audio/T_stt/T_clean/T_state）で構成する。**全部を書き直すのではなく、速度が問題になるホットパスのみ Rust**（LLMは引き続き Ollama HTTP 経由）。実装は子DDで分割し、本DD（親）は全体像・順序・依存・DD-012との境界・定量目標を管理する。

## 背景・課題

- 評価期は Python（faster-whisper / sounddevice）で価値と性能を検証してきた（DD-010: RTF≈0.527, E2E遅延≈4s）。DD-011/DD-012 のライブ経路も Python サイドカーを Rust が中継する形（クロスプロセス境界＝stdout JSON Lines）。
- 製品期の最終形は「whisper-rs / cpal / ONNX Runtime」を Rust に内製し、クロスプロセス境界を**スレッド境界**に置換して低遅延化する（基本設計書 §2・§3）。ここで初めて Rust/低レベル並行処理に本格投資する＝最もリスクが高く最後の段。
- 定量目標が DD-010 実測基準で明確に立つ（下記DoD）ため、CRDT(DD-013)とは別の評価軸を持つ独立親が妥当。

### 2026-06-09
- **見送り（保留）と判断し、親＋子4本（DD-014 / -1 / -2 / -3 / -4）をアーカイブ**（ユーザー判断）。これは「完了」ではなく「着手しない凍結」。理由は3点:
  1. **着手の引き金が未発生**: Python版（DD-010: RTF≈0.527 / E2E≈4s / CPU peak≈30%）で実運用に困っていない。CLAUDE.md 開発方針「速度が問題になったホットパスだけ後から Rust へ移植」に照らし、いまは移植の時期ではない（＝方針どおりの判断であって、断念ではない）。
  2. **調査が否定的結果を出した**: [DD-014-2](DD-014-2_whisper-rs統合とAVX512ビルド検証.md) の予備計測で **素の whisper-rs(Rust)=RTF 3.5 / pywhispercpp=RTF 0.42（約8倍遅い・リアルタイム追従不可）**。AVX-512有効・スレッド数・OpenMP・SIMDフラグ明示の各仮説は潰し済みで、主因は whisper.cpp/ggml の版差＋MSVCコード生成品質の線が濃厚（compute buffer 44.6MB vs 170.2MB が物証）。差を埋めるには **LLVM(約2.5GB)＋clang-cl もしくは OpenBLAS もしくは whisper.cpp 版固定** の重い再検証が必須で、効果も不確実。投資対効果が現時点で見合わない。
  3. **見送っても実機能は無劣化**: 表の機能（録音→文字起こし→要約）は Python版がそのまま製品として動作する。本DDは裏側の速度・CPU改善のための書き直しに過ぎない。
- **再開条件（これが揃うまで再着手しない）**: 実運用で文字起こしの速度／遅延／CPU負荷が*実際の*困りごとになった場合のみ。最初の一手は、隔離スパイク `C:\tmp\whisper-avx512-spike`（本体未汚染で資産流用可）で **clang-cl ビルド or OpenBLAS or whisper.cpp 版固定** により約8倍ギャップが縮むかを再検証 → 縮む見込みが立てば DD-014-1（cpal収音）から再開する。縮まないなら Python サイドカー維持が最適解。
- **副次効果（前進できる点）**: 本DDの見送りにより、DD-012-6（配布パッケージング・Python同梱インストーラ）が抱えていた「Rust移植の様子見で保留」という保留理由が解消される ⇒ Python 同梱前提で配布を前に進められる。
- **成果の保全**: 「実機で AVX-512 が有効化される」確認と「素の whisper-rs は基準比 約8倍遅い」実証は、DD-005 起票時DA#1（『製品ビルド(whisper-rs, AVX-512有効)で実RTFを再測せよ』）への**回答**として価値があるため破棄せず本DD群に保存する。エビデンスはスパイク資産（bench*.log / disasm.txt 等）に残置。
