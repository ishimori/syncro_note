## 目的

[基本設計書（SSOT）](../spec/基本設計書.md)・[要件 File 1〜5](../plan/要件/0_index.md)・[モック画面8枚](../mock/html/) を突き合わせ、**見落としのない SQLite テーブル定義**を確定する。

## 背景・課題

- [要件 File 1 のテーブル定義案](../plan/要件/1_共通仕様.md#L77-L118)（meetings / participants / vocabularies / attachments の4表）は、基本設計書で覆された設計（確定原文 immutable＋LLM整形別レイヤ／話者は confirmed・ai_guess の2層／話者分離は外部スタック）を反映しておらず、**中核テーブル（タイムライン・話者マッピング）が欠落**している。
- 1情報源だけ見て一気に書くと必ず漏れる → **3情報源のトライアンギュレーション＋永続化境界の線引き**を設計手順として固定する。
- ドキュメント間で型がブレている（speaker_id, timestamp, ID型）→ 先に辞書で統一しないと下流が崩れる。

## 設計方針（見落としを潰す順番）

1. **永続化境界を最初に引く** — 会議中の真実源は Rust メモリ（アクター）。SQLite へは batch-flush 対象のみ落とす（[SSOT §2.2/§3.3](../spec/基本設計書.md#L56-L76)）。
2. **画面×データ項目マトリクス** で入力/表示/保存先を網羅 → 「読まれるのに書き込み元が無い」項目をあぶり出す。
3. **ライフサイクル追跡**（scheduled→active→generating→completed＋クラッシュ復旧）で状態 enum と耐久性の漏れを検出。
4. **データ辞書で型・命名を統一** してから論理→物理へ。
5. **SSOT §8 の訂正一覧をDBへ反映**。
