> アプローチ: 標準（決定中心。文書間の食い違いを1枚の辞書に収束させる）

## 目的

ドキュメント間でブレている型・命名・列挙値を1枚のデータ辞書に統一し、論理/物理設計（DD-007-3/4）が一発で確定できる前提を作る。

## 背景・課題

型のブレが地雷。代表例:
- **speaker_id**: 要件 File 3 は `"Speaker_0"`（文字列・名前埋め込み）、[SSOT §3.1](../spec/基本設計書.md#L88-L102) は `u16` ＋**表示名は都度導出（埋め込まない）**。
- **timestamp**: 要件は Unix ms 混在、SSOT は会議内相対 `t_ms` ＋ `seq`。
- **ID型**: meetings は UUID(TEXT)、vocabularies は INTEGER と要件内でも不統一。
- **status**: 要件は scheduled/active/completed の3値。生成中・復旧中が無い。

## 検討内容

決定すべき辞書項目（候補）:
| 項目 | 候補 | 採用方針 |
|------|------|---------|
| 主キーID型 | TEXT(UUID) / INTEGER | エンティティ別に方針決定（外部参照されるものはUUID等） |
| speaker_id | u16相当 INTEGER / 文字列 | SSOT準拠＝整数。表示名はDBに埋め込まず導出 |
| 話者名解決 | confirmed/ai_guess 2層 | confirmed＞ai_guess＞`Speaker_{id}` |
| 時刻 | 絶対(ISO/epoch) vs 相対t_ms | 会議開始=絶対1点、要素は相対t_ms＋seq |
| status enum | 値集合の確定 | scheduled/active/generating/completed/recovering 等を確定 |
| timeline種別 | ai_transcription / human_memo | enum固定 |
| 原文 vs 整形 | 別カラム | 原文 immutable、refined は nullable 別カラム |
| 真偽/フラグ | INTEGER 0/1 | SQLite 慣習に統一 |

## 決定事項

成果物 [データ辞書.md](DD-007-2/データ辞書.md) に集約（[対照表.md](DD-007-2/対照表.md) で食い違いを可視化）。要点（すべてSSOT優先）:
- **speaker_id = INTEGER 仮ID**。`"Speaker_0"` は表示整形。表示名はDBに焼き込まず `confirmed_name ?? ai_guess_name ?? "Speaker_{id}"` で都度導出。
- **時刻**: 絶対は ISO8601 TEXT（scheduled/actual_*）、会議内は相対 `t_ms` INTEGER ＋ `seq`。
- **status enum** = scheduled/active/generating/completed/aborted。
- **原文/整形を別カラム**（text_raw immutable・text_refined 任意・is_refined）。
- 真偽は INTEGER 0/1、子明細IDは INTEGER、実体IDは TEXT(UUID)、列挙は CHECK で二重防御。
- スキーマ版管理はしない（過去データ移行不要・起動時 schema.sql 冪等適用のみ／DD-007-4）。
