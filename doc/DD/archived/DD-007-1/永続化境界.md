> 根拠: [基本設計書 §2.2 / §3.1 / §3.3 / §3.4](../../spec/基本設計書.md)。会議中の真実源は Rust メモリ上のアクター（`SessionState`）。SQLite へは **batch-flush 対象のみ**落とす。

## クラッシュ耐性の設計判断（重要）

[要件 File 4 §3.1](../../plan/要件/4_メモ追記.md#L62-L72) は「強制終了対策に暫定ログ／テンポラリバッファへ即保存」を要求。

- **判断: 専用の暫定ログテーブルは作らない。** `timeline_elements` 自体を WAL 有効SQLiteへ**確定のたびに逐次 flush**（[§3.4](../../spec/基本設計書.md#L153)）することで、これが耐久ログを兼ねる。
- 復旧手順: アプリ起動時に `status='active'` のまま残る会議があれば、`timeline_elements` の flush 済み行から復元できる（会議中の真実源＝確定原文は immutable なので部分復旧で破綻しない）。
- ホットパスは flush の I/O で止めない（別タスク・バッチ flush）。

## メモリ↔DB の対応（flush 方向）

```
SessionState.timeline: Vec<TimelineElement>  ──flush──▶ timeline_elements
SessionState.speaker_map: SpeakerMap          ──flush──▶ speaker_mappings
finish → 清書バッチ出力(Markdown)             ──────────▶ meetings.final_minutes
```

## 🔬 機械検証（照合結果）

SSOT §3.1 の構造体 `AudioChunk / RawSegment / CleanSegment / TimelineElement / SpeakerMap` が全て本表に分類済み（漏れ0）。`TimelineElement` の2バリアント（`AiTranscription` / `HumanMemo`）も `timeline_elements.kind` で表現。
