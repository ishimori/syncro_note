# SynchroniNote（シンクロニ・ノート）

**リアルタイムAI協調型ローカル議事録システム**

会議の音声を、完全オフライン・CPU駆動で「文字起こし → 要約 → 議事録Markdown」まで処理し、最終的には *人間とAIがリアルタイムに同じ議事録を協調編集する* ことを目指すデスクトップアプリケーションです。クラウドへ音声・テキストを一切送信しないため、機密会議でも安心して使えます。

> 🏁 **現在のステータス: Phase 0（評価基盤・性能スパイク）着手中**
> 本リポジトリは現時点では**設計ドキュメントと開発基盤（DD運用）が中心**で、実行可能なアプリはまだありません。
> 「このPCでローカルLLM/STTが実用速度で動くか」をベンチマークで確かめ、以降の設計判断材料にしている段階です。
> 詳細な計画は **[開発ロードマップ](doc/plan/開発ロードマップ.md)** を参照してください。

---

## なぜ作るのか

クラウド型AI議事録（Gemini / Teams / Zoom 等）には2つの課題があります。

1. **事後処理の限界** — 議事録ができるのは会議終了後。進行中にAIの書き起こしを人が直したり、その場で重要事項を追記・共有する「リアルタイムの協調」ができない。
2. **機密情報の漏洩リスク** — クラウド送信型は、インサイダー情報や顧客プライバシーを扱う会議ではセキュリティポリシー上使えないことが多い。

SynchroniNote は **完全ローカル（オフライン）** で動作し、AIによる**話者識別・書き起こし・要約**と、人間による**リアルタイム追記・編集**を同期させることで、この2つを同時に解決します。

詳しい背景・コンセプトは **[企画書](doc/plan/企画書.md)** を参照。

---

## 主な特徴（目指す姿）

- 🔒 **完全オフライン** — 音声もテキストもクラウドに送らない。AI処理はすべてローカルで完結。
- 🧠 **CPU駆動のローカルAI** — GPUなし・大容量RAM環境を前提に、話者識別・STT・LLM要約をCPUで実行。
- 🎙️ **リアルタイム話者識別＋文字起こし** — 「Aさん / Bさん」付きで発話を逐次テキスト化。
- 📝 **コンテキスト追従要約** — 会議の進行に合わせて「現在の要約」を更新し続ける。
- 🤝 **人間×AI 協調エディタ** — AIの自動追記と人間のタイピングが競合しない2ペインUI（CRDTで同期）。

> ⚠️ 上記は企画書の最終形（製品期のゴール）です。現在は中核価値（録音→文字起こし→要約→Markdown）を**バッチCLIで最短検証する**段階から積み上げています。

## アーキテクチャの段階的進化

「初めてのローカルLLM開発をリスクを抑えて評価しながら進める」ため、いきなり最終形を作らず段階的に進化させます。

```
[評価期] Python CLI
  音声ファイル → faster-whisper → (Ollama/Qwen 要約) → Markdown議事録

   ↓ ホットパス移植・リアルタイム要件が固まったら

[製品期] Tauri アプリ（企画書の最終形）
  cpal(Rust) → whisper-rs → diarization(ONNX) → Ollama → Yjs/CRDT 2ペインUI
```

**評価フェーズはPython優先**（回転の速さを重視）、**速度が問題になったホットパスだけRustへ移植**するのが基本方針です。

## 評価指標

ローカルLLM/STTは「動く／動かない」ではなく**速度・精度の実測**で判断します。

| 指標 | 定義 |
|------|------|
| RTF (Real-Time Factor) | 音声長に対する処理時間の比。<1 でリアルタイム可（できれば <0.5） |
| CER / WER | 日本語文字起こしの誤り率 |
| LLM tok/s | Ollama の eval トークン生成速度 |
| E2E 遅延 | 発話 → 議事録反映までの体感遅延（企画書目標は1秒未満、CPUでは要検証） |
| メモリ占有 / スワップ | 同時稼働時のRAM使用とスワップ発生有無（スワップ0を維持） |
| 要約品質ルーブリック | 決定事項/TODO/要点の網羅・正確さを人手評価 |

## 開発の進め方

本プロジェクトは **DD（設計ドキュメント）駆動** で進めます。

1. [開発ロードマップ](doc/plan/開発ロードマップ.md) の各「DD候補」を `/dd new <タイトル>` でチケット化
2. [development-flow.md](doc/development-flow.md) の **5ステップ**に従って実装
3. 実装系DDは検証ステップで上記の評価指標を測定し、DD内ログに数値を残す
4. 完了したら `git commit` → `/dd archive`
5. フェーズ完了ごとにロードマップのステータスを更新

利用できるDDスキル（Claude Code）:

| コマンド | 説明 |
|---------|------|
| `/dd new タイトル` | 新規DD作成 |
| `/dd list` | DD一覧 |
| `/dd log メモ` | ログ追記 |
| `/dd archive 番号` | アーカイブ |
| `/dd search キーワード` | DD検索 |
| `/dd rebuild-index` | インデックス再構築 |

コーディング規約は [coding-standards.md](doc/templates/coding-standards.md)、品質フィルターは [da-method.md](doc/da-method.md) を参照。
