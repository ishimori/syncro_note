# SynchroniNote（シンクロニ・ノート）

**完全オフライン・CPU駆動で動く、リアルタイムAI協調型ローカル議事録システム**

> 🔒 クラウドに音声もテキストも一切送らない &nbsp;|&nbsp; 🧠 STT・要約・話者識別をすべてローカルAIで &nbsp;|&nbsp; 🤖 開発は **Claude Code** ＋ 自作の **DD（設計ドキュメント）駆動** で進行中

会議の音声を「文字起こし → 要約 → 議事録Markdown」まで**ネットにつながずに**処理し、最終的には *人間とAIがリアルタイムに同じ議事録を協調編集する* ことを目指すデスクトップアプリです。機密会議でも安心して使えることを存在意義にしています。

## なぜ作るのか

クラウド型AI議事録（Gemini / Teams / Zoom 等）には2つの課題があります。

1. **事後処理の限界** — 議事録ができるのは会議終了後。進行中にAIの書き起こしを人が直したり、その場で重要事項を追記・共有する「リアルタイムの協調」ができない。
2. **機密情報の漏洩リスク** — クラウド送信型は、インサイダー情報や顧客プライバシーを扱う会議ではセキュリティポリシー上使えないことが多い。

SynchroniNote は **完全ローカル（オフライン）** で動作し、AIによる**話者識別・書き起こし・要約**と、人間による**リアルタイム追記・編集**を同期させることで、この2つを同時に解決します。

詳しい背景・コンセプトは **[企画書](doc/plan/企画書.md)** を参照。

## 主な機能

会議の前→中→後を1本の流れでカバーします（画面 S-01〜S-08）。

- 🗓️ **カレンダー＆会議管理** — 予定の作成・編集・移動・削除。カレンダー予定の**コピペ取込**（テキストを貼るとLLMが構造化して会議登録）。
- 📎 **事前資料の取込** — Excel / PDF を添付してテキスト抽出、要約の文脈として利用。
- 🎙️ **プリフライト** — 録音前に入力レベル・無音をチェックし、収音デバイスを選択。
- ⚡ **準リアルタイム文字起こし** — マイク入力を 8〜12 秒の VAD 区切りで逐次STT。確定テキストを即表示し、LLMが**非ブロッキングで追い上げ整形**。
- 🧑‍🤝‍🧑 **話者識別** — CAM++（ONNX）話者埋め込み＋クラスタリングで「話者A / 話者B」を付与、色分け表示。
- 📝 **議事録の自動生成** — 会議終了後にバッチで要約・決定事項・TODO を清書し、Markdown としてプレビュー＆保存。
- ⚙️ **設定** — モデルやデバイスなどの設定を永続化し、実処理へ反映。

## 完全ローカルAIスタック

すべてのAI処理がこのPCの中で完結します（クラウド送信なし）。実測で選定したモデルは以下のとおり。

| 役割 | 採用 | 備考（実測で決定） |
|------|------|------|
| 文字起こし（STT） | faster-whisper / whisper.cpp **medium** ＋ VAD | VADで無音を除くと日本語CERが約5倍改善（無音幻覚が主因と判明） |
| ライブ整形（追い上げ） | Ollama **qwen3:8b** | TTFT最速・日本語固有名詞の補正に強い |
| 終了後バッチ清書 | Ollama **gemma4:26b**（MoE・活性4B） | 「26B級の品質を4B級の速度で」を本機で実証 |
| 話者識別 | **CAM++ ONNX** 埋め込み → KMeans | モデル未配置時は MFCC+KMeans の完全オフラインへフォールバック |

> 数値の根拠は各DDの結果ログ（[DD-002](doc/DD/) 要約速度 / [DD-003](doc/DD/) STT実機 / [DD-004-1](doc/DD/) 話者分離 / [DD-005](doc/DD/) whisper.cpp）と [基本設計書](doc/spec/基本設計書.md) に残しています。

## GPUなしPCで動かす — 現実とトレードオフ

このプロジェクトの裏テーマは **「GPUを積んでいないPCで、ローカルLLM議事録はどこまで実用になるか」** です。結論を正直に書きます。

> **動く。ただしメモリは必須で、結局GPUがないと苦しい。**

- ✅ **CPUでも動く** — AVX-512 を活かせば、文字起こし（whisper medium）は **RTF 0.43〜0.61（<1＝リアルタイム可）**、要約は実用速度に乗りました。
- ⚠️ **メモリが効く** — 総RAMは59.7GBあっても、ブラウザ/エディタ等の常駐で**実際に使えるのは実測 約9.6GB**。**32Bクラスの要約モデル（20〜24GB）はそのままでは載りません**。本機では「26B級MoE（活性4B）」で品質と省メモリを両立させています。
- ⚠️ **同時実行が鬼門** — STT と LLM を無対策で同時に回すと CPU を奪い合い、whisper の RTF が 0.61 → 1.48 に悪化（スラッシング）。**時間分離**（会議中はSTTに集中し、重い要約は終了後に回す）で回避しています。
- 🔮 **iGPU は将来の伸びしろ** — 内蔵GPU（Radeon 780M）への llama.cpp/whisper の Vulkan 部分オフロードは候補ですが、RAM共有のため過度な期待は禁物。
- 🧱 **だからこそ Tauri** — Electron と違い OS標準のWebViewを借りる Tauri は省メモリ。**限られたRAMの取り分をローカルLLMに回せる**のが本機では決定的でした（理由の詳細は [ロードマップ §7.1](doc/plan/開発ロードマップ.md)）。

要するに——**GPUがあれば何も悩まない領域を、CPU＋大容量RAM＋設計の工夫でなんとか実用圏に引き上げている**プロジェクトです。

## Claude Code × DD駆動開発

本プロジェクトは [Claude Code](https://claude.com/claude-code) と協働し、**DD（設計ドキュメント）駆動**で開発しています。

**DDとは**: 1つの作業を始める前に「何を・なぜ・どう作り・何をもって完了とするか」を1枚の設計ドキュメント（＝チケット）にまとめ、実装後に**実測値・決定理由・完了確認を同じ文書へ残す**運用です。`.claude/skills/` に置いた自作の `/dd` スキルで回します。

```text
/dd new <タイトル>   →  設計ドキュメント（チケット）を起こす
   ↓ 実装（開発5ステップ: doc/development-flow.md）
   ↓ 検証（評価指標 RTF / CER / tok/s 等を実測しログに記録）
git commit  →  /dd archive   →  DD索引を再生成
```

| コマンド | 説明 |
|---------|------|
| `/dd new タイトル` | 新規DD作成 |
| `/dd list` | DD一覧 |
| `/dd log メモ` | ログ追記 |
| `/dd archive 番号` | アーカイブ |
| `/dd search キーワード` | DD検索 |
| `/dd rebuild-index` | インデックス再構築 |

この運用のおかげで、**「なぜこのモデルを選んだか」「なぜ32Bを諦めたか」といった判断の根拠がすべて [DD索引](doc/DD/DD-INDEX.md) から辿れます**。設計のSSOT（唯一の正）は [基本設計書](doc/spec/基本設計書.md) と [DB設計](doc/spec/db/)、品質フィルターは [da-method.md](doc/da-method.md) に集約しています。

## TauriアプリのUI検証（Playwright / CDP）

Tauriアプリの画面は、**スクリーンショットを人間に頼まず、Claude 自身が確認**します。2段構えです。

1. **見た目の確認 — Playwright（MCP）**: `tauri dev` が配信する `http://localhost:1420` を Playwright で開き、リサイズ→スクショ→目視。素早いUIレビュー用。
2. **本物のE2E動作確認 — CDP 直結**: `scripts/start-app.sh` で起動した**実ウィンドウ**には WebView2 のデバッグ窓口（CDP）が開きます。そこへ `scripts/tauri-cdp.mjs` で直結すると、画面操作だけでなく **`invoke` → Rust → 実SQLite** まで本物のまま叩けます。

```bash
node scripts/tauri-cdp.mjs snapshot                       # 操作可能な要素を読む
node scripts/tauri-cdp.mjs click "<セレクタ>"              # Vue/Quasarのハンドラを発火
node scripts/tauri-cdp.mjs invoke list_meetings '{"year":2026,"month":6}'  # Rustコマンドを直接実行
node scripts/tauri-cdp.mjs shot out.png                   # 実ウィンドウをPNG出力
```

これにより「見た目だけ」を超えて、**Rust連携・実DB込みのエンドツーエンド挙動を Claude が自動で検証**できます。Python側のロジックは [pytest](python/tests/)（VADチャンク化・話者分離・要約・抽出など14本）でカバーしています。

## アーキテクチャの段階的進化

「初めてのローカルLLM開発をリスクを抑えて進める」ため、いきなり最終形を作らず段階的に進化させます。

```text
[評価期] Python CLI / ハーネス
  音声ファイル → faster-whisper → (Ollama 要約) → Markdown議事録

   ↓ 価値とモデル性能を実測で検証してから

[製品期] Tauri アプリ（企画書の最終形）
  cpal(Rust) → whisper-rs → diarization(ONNX) → Ollama → Yjs/CRDT 2ペインUI
```

**評価フェーズはPython優先**（回転の速さを重視）、**速度が問題になったホットパスだけRustへ移植**するのが基本方針です。

### ロードマップ進捗

| フェーズ | 目的 | 状態 |
|---------|------|------|
| **Phase 0** 評価基盤・性能スパイク | ローカルLLM/STTが実機で実用速度か数字で確認 | ✅ 完了 |
| **Phase 1** バッチMVP | 音声ファイル → 議事録Markdown | ✅ 一次達成（DD-008） |
| **Phase 2** 準リアルタイム化 | マイク入力からの逐次処理 | ✅ 達成（DD-010, RTF 0.527） |
| **Phase 3** 話者識別（Diarization） | 「話者A/B」付きの文字起こし | ✅ 採用確定（DD-004-1, CAM++ ONNX） |
| **Phase 4** 協調UI（Tauri/Yjs/CRDT）＋Rust移植 | 2ペイン協調エディタ・低遅延部のRust化 | 🚧 進行中（UI骨格✅／CRDT・Rust移植中） |
| **Phase 5** チューニング・実証実験 | メモリ最適化・実会議UX評価 | ⏳ 計画 |

各フェーズのDD候補・完了条件(DoD)・評価指標は [開発ロードマップ](doc/plan/開発ロードマップ.md) を参照。
